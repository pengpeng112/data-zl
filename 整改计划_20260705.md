# 数据资产平台整改执行计划

## 1. 当前核查结论

本仓库是实际程序代码，包含 `backend/`、`frontend/`、`deploy/` 三部分。当前不是 Git 仓库，`deploy/` 目录为空。

已验证：

- 后端：`backend\.venv\Scripts\pytest.exe tests -q` 通过，结果 `180 passed`。
- 前端构建：`frontend\npm.cmd run build` 通过。
- 前端类型检查：`frontend\npm.cmd run typecheck` 在脱离沙箱后通过；沙箱内失败是因为不能写 `node_modules\.vue-global-types`。

阶段判断：

- 前端一级模块已补齐，不再是 `28` 文档中“只看到数据资产页面”的状态。
- 后端模型、路由、测试骨架较完整，但多处仍是“本地元数据/占位执行/登记任务”，未形成真实数据源采集、真实同步、真实运维执行闭环。
- 部署交付缺失，无法直接按内网 8.83 目标环境落地。

## 2. 最高优先级问题

### P0-1 代码目录缺少版本管理

现状：`F:\python\data-asset-platform` 执行 `git status` 返回 `not a git repository`。

影响：后续 AI 无法稳定审阅 diff、回滚变更、按提交拆分任务。

整改：

1. 在开始功能整改前初始化 Git，或确认外层是否有真实仓库。
2. 首次提交前确认 `.gitignore` 已排除 `.env`、`.venv/`、`node_modules/`、`dist/`、`__pycache__/`、日志。
3. 每完成一个阶段单独提交。

验收：`git status --short` 可用，工作区能展示后续 AI 的改动。

### P0-2 部署目录为空

现状：`deploy/` 没有文件。

影响：代码能构建，但不能形成可复现的内网部署包。

整改：

1. 新增 `deploy/README.md`，说明 8.83 离线部署步骤。
2. 新增 `deploy/backend.env.example`、`deploy/frontend.env.example`。
3. 新增 `deploy/docker-compose.yml` 或 systemd/nginx 方案二选一；如暂不使用 Docker，则提供 `data-asset-backend.service`、`nginx.conf`。
4. 明确 PostgreSQL、后端端口、前端静态目录、反向代理路径、日志目录。

验收：按 `deploy/README.md` 可从空目录部署到可访问状态。

## 3. 后端未完成项

### P1-1 元数据采集未真正连接源库

现状：

- `backend/app/api/v1/metadata_changes.py` 的 `collect_metadata()` 只从本地 `AssetTable/AssetColumn` 生成快照。
- `backend/app/services/metadata_collector.py` 有采集适配器，但没有接入 `collect_metadata()`。

整改：

1. 在 `collect_metadata()` 中按 `AssetDataSource` 创建 `DB_CONNECTOR_MAP` 连接器。
2. 使用 `METADATA_COLLECTOR_MAP` 采集 schema/table/column。
3. 写入 `AssetMetadataColumnSnapshot`，并更新 `MetadataSnapshot.table_count/column_count`。
4. 对 Oracle 11g 遵守 `ROWNUM <= N`，禁止全表扫描。
5. 凭据只通过 `credential_ref` 或环境变量解析，不能写入库或日志。

验收：

- 不存在的 `source_code` 返回 400/404。
- 注册但连不通的数据源写入失败任务记录。
- 注册且可连的数据源能生成非空快照。

### P1-2 同步执行器仍是 stub

现状：`backend/app/services/sync_executor.py` 对 `identity_department`、`identity_person`、`medical_code`、`metadata_collect` 返回 `status="stub"`。

整改：

1. 定义执行器接口：`execute(db, source, target, params) -> SyncResult`。
2. 将 `identity_department/person` 接到人员科室采集与差异比对。
3. 将 `medical_code` 接到诊断/手术字典采集与差异生成。
4. 将 `metadata_collect` 接到 P1-1 的真实元数据采集。
5. 所有执行结果写 `asset_govern_audit_logs`，失败也要记录异常摘要。

验收：调用同步接口不再返回 `stub`，并能产生可查询的差异或快照。

### P1-3 人员科室采集只是登记任务

现状：`backend/app/api/v1/identity.py` 的 `POST /api/v1/identity/collect-sources` 仅返回 `scheduled`。

整改：

1. 实现 HIS/HRP 只读采集适配器。
2. 写入或更新 `IdentityDepartment`、`IdentityPerson`、`IdentityPersonSource`、`IdentityAccount`。
3. 生成 `IdentitySyncDiff`，覆盖新增、缺失、字段不一致、临时人员、账号未绑定。
4. 前端采集按钮展示任务结果和差异数量。

验收：执行采集后 `identity/sync-diffs` 有真实 diff 数据或明确的无差异结果。

### P1-4 医疗字典差异采集未闭环

现状：`backend/app/api/v1/dict_medical_api.py` 已有编码体系、编码项、映射、差异列表 API，但没有 HIS/EMR 字典采集入口。

整改：

1. 新增 `POST /api/v1/dict-medical/collect-sources`。
2. 采集诊断、手术、科室相关字典，写入 `DictMedicalCodeSet/DictMedicalCodeItem`。
3. 生成 `DictMedicalSyncDiff`。
4. 前端 `dict/sync-diffs` 增加“采集/比对”按钮。

验收：采集后可按 `status=open` 查询差异；重复采集不产生重复脏数据。

### P1-5 运维工具执行是假执行

现状：`backend/app/api/v1/ops_tools.py` 的 `execute_run()` 只把状态改成 `executed`，`execution_summary` 是占位文本。

整改：

1. 根据 `OpsToolTemplate.execution_mode` 分发执行器。
2. 支持 `readonly_sql`、`stored_procedure`、`http_api`、`sync_executor`，默认禁用写操作。
3. `require_second_confirm=True` 时必须提交二次确认参数。
4. 执行前检查工具白名单、审批状态、操作者、风险等级。
5. `input_params_masked` 入库前脱敏。

验收：未审批不能执行；缺二次确认不能执行；执行结果含真实状态、耗时、影响范围。

### P1-6 APScheduler 只是记录任务

现状：`backend/app/main.py` 启动 APScheduler 后，`_run_metadata_collect()` 仅写一条 `SchedulerJob` 成功记录，不触发真实采集。

整改：

1. 将 `_run_metadata_collect(source_code)` 调用 P1-1 的采集服务。
2. 支持 `metadata_scan`、`quality_check`、`sync_executor`、`data_lifecycle` 四类任务。
3. 解析 cron 时使用 `APP_SCHEDULER_TIMEZONE`。
4. 增加 scheduler 单元测试，避免应用导入时重复注册任务。

验收：创建 scheduled job 后，启动服务能执行真实任务并写入结果。

### P1-7 数据质量仍主要依赖本地元数据

现状：`backend/app/api/v1/quality.py` 中 `SOURCE_CONNECTIVITY` 只读取 `last_check_status`，没有触发连接检测；孤儿率也只读 `validation_metrics`。

整改：

1. `SOURCE_CONNECTIVITY` 可选择实时调用 `check_source()` 或标记过期状态。
2. 增加小样本孤儿率规则，仅对白名单关系执行源库采样。
3. 大表必须要求采样条件，`HIS.LAB_RESULT` 只能通过 `TEST_NO` 子查询限定。
4. 质量任务记录采样 SQL 摘要，不记录敏感值。

验收：质量报告能区分“未检测”“最近失败”“实时失败”“采样异常”。

## 4. 前端整改项

### P2-1 关系复核工作台功能偏弱

现状：`frontend/src/views/asset/relation-review/index.vue` 已存在，但只编辑 `AssetRelation` 部分字段，没有完整工作流。

整改：

1. 增加筛选：系统、数据源、schema、置信度、验证状态、关键字。
2. 增加 approve/reject 操作，对接 `/api/v1/relations/{id}/review`。
3. 增加字段映射视图，对接 `/api/v1/relations/field-mappings`。
4. 展示审计记录入口。

验收：人工可完成“筛选候选关系 -> 修改说明 -> 通过/拒绝 -> 查看审计”。

### P2-2 数据源页面缺少真实采集闭环

现状：`frontend/src/views/asset/sources/index.vue` 有连通性检测和资产树，但采集/调度/最近快照不完整。

整改：

1. 增加“采集元数据”按钮，对接 `/api/v1/sources/{source_code}/collect-metadata`。
2. 展示最近快照、表数、字段数、最近检测状态。
3. 采集失败时展示后端错误摘要，不吞异常。

验收：页面能从数据源完成检测、采集、查看快照三步。

### P2-3 运维、人员、字典、变更页面需要端到端校验

现状：页面和路由已存在，构建通过，但仍需按真实接口逐页验收。

整改：

1. 为 `views/ops/*` 补齐执行结果、二次确认、审计详情。
2. 为 `views/identity/*` 补齐采集入口、差异处理、人员画像跳转。
3. 为 `views/dict/*` 补齐采集入口、映射审核、差异处理。
4. 为 `views/metadata-changes/*` 补齐影响分析、状态流转、负责人分配。

验收：每个一级菜单至少完成列表、详情/编辑、状态动作、错误提示。

## 5. 安全与治理

### P3-1 RBAC 未真正拦截接口

现状：`backend/app/main.py` 定义了 `ROLE_REQUIRED`，但中间件只校验 Bearer Token，没有按角色权限拒绝访问。

整改：

1. `ApiKey` 需要能映射 `user_identifier`。
2. 中间件读取 `asset_user_roles`、`asset_role_permissions`。
3. 按路径前缀和 HTTP 方法映射 resource/action。
4. 保留平台管理员绕过能力。

验收：无权限 token 访问 `/api/v1/ops/*` 返回 403。

### P3-2 Token 与凭据安全仍不完整

现状：已有 `expires_at` 字段和 `credential_encrypt_key` 配置，但需要全链路使用。

整改：

1. `.env.example` 补充 `APP_CREDENTIAL_ENCRYPT_KEY`、`APP_SCHEDULER_TIMEZONE`、retention 配置。
2. 所有源库账号密码使用密钥引用解析，禁止直接从接口写明文。
3. 审计日志、运维参数、异常信息统一脱敏。
4. API Token 过期测试覆盖启用、禁用、过期三类。

验收：日志和数据库中不出现身份证、电话、地址、明文密码。

### P3-3 请求限流未接入

现状：`requirements.txt` 包含 `slowapi`，但应用未配置限流中间件。

整改：

1. 接入 `slowapi` 或等价限流。
2. 对登录/初始化、AI 工具、运维执行、采集触发接口设置更严格限制。
3. 限流错误返回统一 `{code, message, data}`。

验收：短时间高频调用敏感接口返回 429。
