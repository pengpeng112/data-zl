# 医院数据库智能分析平台执行方案

## 1. 建设背景

医院内部存在大量异构业务系统，包括 HIS、EMR、LIS、PACS、手麻、病理、医保、集成平台、数据中心等。数据库类型可能包含 Oracle、SQL Server、PostgreSQL、MySQL 及国产数据库。各系统表结构复杂、字段含义不统一、显式外键缺失、历史 SQL 分散，导致信息科在取数、写视图、接口治理、科研数据支持、数据质量核查时依赖人工经验。

本方案目标是建设一套内网部署的“医院数据库智能分析平台”，实现自动探库、元数据画像、关系梳理、字典变更检测、AI 辅助解析、SQL/视图生成、关系图展示和报告输出。

## 2. 建设目标

### 2.1 总体目标

构建一个面向医院信息科的数据资产智能分析平台，将数据库结构、字段含义、数据分布、表间关系、字典变化、历史 SQL、接口规范和人工经验沉淀为可检索、可展示、可被 AI 调用的知识底座。

### 2.2 核心能力

- 自动分析数据库结构，包括库、schema、表、字段、索引、约束、视图。
- 自动生成表画像、字段画像、数据质量画像。
- 自动推断表间关系和字段关联路径。
- 自动检测表结构、字段、字典、数据分布的周期性变化。
- 支持 Dify 内网大模型工作流，对元数据进行解释、归纳和建议生成。
- 支持 AI 基于已确认关系生成 SQL、视图、数据质量检查语句和分析报告。
- 支持关系图、血缘图、字段链路图和业务实体图展示。
- 支持多数据库、多系统、多医院扩展。

## 3. 设计原则

### 3.1 安全优先

- 所有数据库连接使用只读账号。
- AI 不直接裸连数据库。
- SQL 执行必须经过只读沙箱校验。
- 敏感字段自动识别和脱敏。
- 全部探库、分析、SQL 生成、执行行为保留审计日志。

### 3.2 内网优先

- 探库服务、元数据库、向量知识库、Dify、图数据库均部署在医院内网。
- 若临时使用外网大模型，只允许发送脱敏后的元数据摘要，不发送真实患者数据。
- 架构预留模型供应商切换能力，后续可从外网模型平滑迁移到内网 Dify 或私有化大模型。

### 3.3 可扩展

- 数据源连接器插件化。
- 关系推断规则插件化。
- 医疗业务实体映射可配置。
- 多医院多租户模型从第一版开始预留。

### 3.4 人机协同

- 程序负责采集、统计、比对和校验。
- AI 负责解释、归纳、生成建议和 SQL 草稿。
- 人工负责确认关键关系、字段含义、业务口径和生产使用。

### 5.3.1 元数据库选型：PostgreSQL 与 Oracle 对比

医院本地已有 Oracle 数据库，可以作为业务库、接口库或结果库继续使用。但本平台的“元数据库”建议优先使用 PostgreSQL，不建议直接放在 Oracle 中，原因如下。

| 维度 | PostgreSQL | Oracle |
|---|---|---|
| JSON 支持 | JSONB 原生强，适合存画像、规则、AI 结果 | 支持 JSON，但开发便利性弱一些 |
| 向量扩展 | pgvector 成熟，MVP 可直接复用 | 原生向量能力依赖版本，部署不一定方便 |
| 开发成本 | Python/FastAPI 生态适配简单 | 连接、驱动、授权、对象管理相对重 |
| 部署成本 | 开源，Docker 部署方便 | 授权、实例维护成本高 |
| 可移植性 | 适合多医院复制部署 | 每院 Oracle 版本和权限差异较大 |
| 图扩展 | 可用 AGE 或关系边表 | 可做关系表，但图能力不如专用图谱方案 |
| 规则配置 | JSONB + GIN 索引适合复杂规则 | 也可实现，但维护成本更高 |
| AI 结果入库 | 适合存非结构化和半结构化结果 | 可实现，但不如 PostgreSQL 灵活 |

推荐结论：

```text
业务源库：继续使用各系统原数据库，不改动。
平台元数据库：建议使用 PostgreSQL。
本地 Oracle：可作为被探查的数据源，也可作为部分结果同步目标，但不建议作为第一版平台元数据库。
```

如果医院不希望新增 PostgreSQL，也可以使用 Oracle 做第一版元数据库，但需要牺牲以下便利性：

- pgvector 不能直接使用。
- JSON 规则配置查询复杂度更高。
- Docker 化复制到其他医院更麻烦。
- 后续 AI 结果、规则配置、半结构化画像维护成本更高。

因此本方案建议保持：

```text
PostgreSQL = 平台元数据库 + 规则中心 + 快照中心 + AI结果中心
Oracle/SQL Server/PostgreSQL = 被探查的业务数据库
```

### 5.4 AI

| 能力 | 推荐方式 | 说明 |
|---|---|---|
| 内网 AI 应用 | Dify Workflow | 已部署，可作为 AI 分析执行器 |
| 本地模型 | 32B 大模型 | 足够支撑第一版元数据解释、关系说明、SQL 草稿 |
| RAG | Dify 知识库 / 自研检索 + pgvector | 文档、SQL、报告检索 |
| Agent 工具调用 | 自研 Tool API + Dify 调用 | 核心工具和权限由平台控制 |

### 5.4.1 外网大模型与内网 Dify 的分工

可以采用“外网大模型首轮深度探索 + 内网 Dify 日常持续梳理”的双模型策略，但必须控制数据边界。

推荐分工：

| 阶段 | 模型 | 输入 | 输出 | 说明 |
|---|---|---|---|---|
| 首轮探索 | 外网强模型 | 脱敏元数据摘要、字段名、注释、统计指标、脱敏样本 | 初版业务域、实体映射、关系候选、规则建议 | 用强模型快速建立第一版知识底座 |
| 日常维护 | 内网 Dify + 32B 模型 | 新增表字段、字典变化、画像变化、已确认关系 | 变化解释、影响分析、规则建议、报告草稿 | 数据不出内网，适合长期运行 |
| SQL 生成 | 内网 Dify 优先 | 已确认关系、字段画像、需求描述 | SQL 草稿、视图草稿、检查 SQL | 必须经过 SQL 沙箱和人工复核 |
| 复杂问题复核 | 外网强模型可选 | 去敏后的 SQL、元数据、问题描述 | 第二意见 | 仅用于复杂关系或设计问题，不直接接触真实数据 |

外网大模型只允许接收：

```text
表名、字段名、字段类型、字段注释、行数、空值率、唯一率、top枚举脱敏值、字段语义摘要、历史SQL去敏版本
```

禁止发送：

```text
患者姓名、身份证、电话、地址、病历文本原文、检查报告原文、检验结果明细原文、数据库连接信息、账号密码
```

推荐流程：

```text
第一次建设：
内网探库 -> 生成脱敏元数据包 -> 外网强模型分析 -> 结果导入元数据库 -> 人工确认 -> 内网Dify接管日常

日常运行：
定期探库 -> 快照diff -> 内网Dify分析变化 -> 写入变更事件 -> 前端展示 -> 人工确认
```

这样可以兼顾首轮建设效率和长期安全性。

- 支持新增、编辑、禁用数据源。

- 支持连接测试。
- 支持配置只读账号、schema 范围、排除表、采样行数、探库频率。

#### 数据源字段

```text
schema_include
schema_exclude

```

### 6.2 自动探库

#### 采集内容

- 数据库、schema、表、字段。
- 表注释、字段注释。
- 主键、唯一约束、外键、索引。
- 视图定义、依赖对象。
- 表行数。
- 字段空值率、唯一率、最大长度。
- 枚举字段 top N 值。
- 时间字段最小值、最大值、月度分布。
- 脱敏样本值。

#### 探库策略

- 小表可全量统计。
- 大表使用采样或分段统计。
- 字段画像按重要性分级执行。
- 核心表每天画像，普通表每周画像。
- 大字段、CLOB、BLOB 默认不采样原文，只记录长度和是否存在。

### 6.3 表字段画像

#### 表画像

- 表名、中文名、所属系统。
- 行数、时间范围、最近探库时间。
- 疑似业务域：患者、住院、门诊、诊断、手术、检查、检验、费用、字典。
- 主键候选。
- 关联表候选。
- 数据质量概览。

#### 字段画像

- 字段名、类型、长度、可空。
- 注释、AI 解释、人工解释。
- 空值率、唯一率、最大长度。
- top 枚举值。
- 敏感字段识别。
- 疑似语义类型：患者ID、住院号、门诊号、检查号、检验单号、医生编码、科室编码、时间字段。

### 6.4 关系推断

#### 关系来源

| 来源 | 说明 | 初始置信度 |
|---|---|---:|
| 数据库外键 | 显式 FK | 1.00 |
| 字段名一致 | PATIENT_ID、VISIT_ID、TEST_NO 等 | 0.70 |
| 值覆盖率 | A 字段值大比例存在于 B 字段 | 0.80 |
| 联合键匹配 | PATIENT_ID + VISIT_ID 等 | 0.85 |
| 历史 SQL | 旧视图/报表中存在 join | 0.90 |
| AI 推断 | 基于字段名、注释、样本推断 | 0.40-0.70 |
| 人工确认 | 信息科确认 | 1.00 |

#### 常见医疗关系模板

### 6.5 字典与结构变更检测

#### 检测机制

每次探库生成快照，然后与上一次快照做 diff。

```text
snapshot_t1
snapshot_t2
  │
  ├── 表变化
  ├── 字段变化
  ├── 索引/约束变化
  ├── 字典项变化
  └── 数据分布变化
```

#### 检测内容

- 新增表、删除表、表注释变化。
- 新增字段、删除字段、字段类型变化、长度变化、可空变化。
- 主键、索引、外键变化。
- 字典新增项、停用项、名称变化、编码变化。
- 行数突增突降。
- 空值率异常升高。
- 枚举值新增或消失。

#### 字典重点表

```text
科室字典
人员字典
诊断字典
手术字典
检查项目字典
检验项目字典
收费项目字典
医保目录
平台上报值域
```

#### AI 影响分析

当检测到变更后，平台自动推送 Dify 生成影响分析。

示例：

```text
检测到 HIS.DEPT_DICT 新增 020111 耳外科。
可能影响：
1. 以耳科为口径的科研取数 SQL。
2. 出院科室统计报表。
3. 三甲上报科室映射。
4. 旧 SQL 中 WHERE DEPT_DISCHARGE_FROM='020102' 可能漏数。
建议：将相关 SQL 调整为 IN ('020102','020101','020111')，并人工确认。
```

### 6.6 数据质量规则配置中心

数据质量规则配置中心用于对表、字段、字典、值域、数据完整性、合理范围、字段有效性进行持续质控。规则应支持前端可视化维护、启停、版本管理、执行调度、结果追踪和人工处理闭环。

#### 6.6.1 规则目标

- 判断字段值域是否准确。
- 判断字段内容是否缺失。
- 判断数值、日期、长度范围是否正常。
- 判断编码是否存在于标准字典或院内字典。
- 判断字段是否长期为空、长期固定值、疑似无用。
- 判断主键、唯一性、外键、业务关联是否异常。
- 判断数据变化趋势是否异常。
- 判断视图或接口字段是否满足上报规范。

#### 6.6.2 规则分类

| 规则类别 | 示例 | 适用对象 |
|---|---|---|
| 完整性规则 | 姓名、患者ID、住院号、入院时间不能为空 | 字段 |
| 唯一性规则 | PATIENT_ID + VISIT_ID 在住院主表唯一 | 表/字段组 |
| 值域规则 | 性别只能为男/女/未知；离院方式必须在字典内 | 字段 |
| 字典一致性规则 | DEPT_CODE 必须存在于 DEPT_DICT | 字段关系 |
| 范围规则 | 年龄 0-130；体温 30-45；血压不超过合理范围 | 数值字段 |
| 日期规则 | 出院时间不得早于入院时间；报告时间不得早于申请时间 | 日期字段 |
| 长度规则 | 身份证号长度应为 18；手机号长度 11 | 文本字段 |
| 格式规则 | 身份证、手机号、医保号、日期字符串格式 | 文本字段 |
| 枚举稳定性规则 | 枚举值突然新增、消失或占比异常 | 枚举字段 |
| 关联完整性规则 | LAB_RESULT.TEST_NO 必须能关联 LAB_TEST_MASTER.TEST_NO | 表关系 |
| 空值率规则 | 核心字段空值率超过阈值告警 | 字段 |
| 重复率规则 | 主键候选重复率超过阈值告警 | 字段组 |
| 异常波动规则 | 当日数据量较近 30 日均值变化超过 3 倍 | 表 |
| 字段无用判定 | 字段长期 100% 空、长期单一值、从未被 SQL/视图引用 | 字段 |
| 上报规范规则 | 省平台必填字段不能为空；值域必须符合标准 | 视图/接口 |

#### 6.6.3 值域准确性规则

值域准确性规则用于判断字段值是否在合法集合中。

规则来源：

- 数据库字典表，例如 `HIS.DEPT_DICT`、`HIS.STAFF_DICT`、`HIS.DIAGNOSIS_DICT`。
- 平台标准值域，例如性别、民族、婚姻、科室、离院方式。
- 本地人工维护值域。
- Excel/CSV/Word/PDF 接口规范解析后的值域。

示例规则：

```text
规则名称：出院科室编码合法性
规则对象：HIS.PAT_VISIT.DEPT_DISCHARGE_FROM
规则类型：字典一致性
字典来源：HIS.DEPT_DICT.DEPT_CODE
通过条件：DEPT_DISCHARGE_FROM in (select DEPT_CODE from HIS.DEPT_DICT)
严重级别：高
执行频率：每日
```

值域质控输出：

```text
字段：DEPT_DISCHARGE_FROM
总行数：574979
非法值数量：23
非法值样本：['020999', 'OLD01']
影响表：PAT_VISIT
建议：确认是否历史科室或字典缺失。
```

#### 6.6.4 内容缺失规则

内容缺失规则用于识别关键字段缺失、记录缺失和关联数据缺失。

缺失类型：

- 字段为空：字段值为 NULL 或空字符串。
- 关联缺失：主表有记录，子表无对应记录。
- 时间段缺失：某天、某月无数据。
- 报告缺失：有检查申请但无检查报告。
- 结果缺失：有检验主记录但无检验明细。

示例：

```text
规则名称：检查申请必须有报告
主表：HIS.EXAM_MASTER
子表：HIS.EXAM_REPORT
关联字段：EXAM_NO
异常条件：EXAM_MASTER.EXAM_NO not exists in EXAM_REPORT.EXAM_NO
严重级别：中
```

#### 6.6.5 范围合理性规则

范围规则用于检查数值、日期、长度和业务区间是否合理。

示例：

```text
年龄：0 <= AGE <= 130
体温：30 <= TEMPERATURE <= 45
脉搏：20 <= PULSE <= 250
收缩压：50 <= SBP <= 260
舒张压：30 <= DBP <= 160
住院天数：0 <= DISCHARGE_DATE_TIME - ADMISSION_DATE_TIME <= 3650
报告时间：REPORT_TIME >= APPLY_TIME
手术结束时间：END_DATE_TIME >= START_DATE_TIME
```

范围规则应支持：

- 固定阈值。
- 按科室、年龄段、性别配置不同阈值。
- 按历史分布自动生成建议阈值。
- AI 根据字段含义推荐初始阈值，人工确认后启用。

#### 6.6.6 字段无用判定规则

字段无用不能只靠空值率，需要综合判断。

判定指标：

- 长期空值率 100%。
- 长期只有一个固定值。
- 最近 N 次探库分布无变化。
- 未被任何视图、报表、接口 SQL 引用。
- 字段注释为空且 AI 无法识别明确含义。
- 字段名包含 `RESERVED`、`FIELD1`、`BZ1`、`TMP`、`BAK` 等疑似保留字段。

无用字段状态不应自动删除，只做标记：

```text
有效字段
疑似无用
历史保留
待人工确认
确认无用
```

### 6.7 Dify 集成

#### Dify 定位

Dify 作为内网 AI 分析执行器，不作为主元数据库，不直接连接业务数据库。

#### 调用方式

- 平台生成 AI 分析任务。
- 任务切片后推送 Dify Workflow。
- Dify 返回 JSON 结构化结果。
- 平台校验 JSON 后写回元数据库。

#### 推荐 Workflow

| Workflow | 输入 | 输出 |
|---|---|---|
| 单表画像解析 | 表结构、字段画像、样本 | 表含义、业务域、主键候选、字段解释 |
| 表关系推断 | 两张表画像、值覆盖率、字段相似度 | 关系字段、置信度、理由 |
| 字典变化影响分析 | 变更项、依赖对象 | 影响范围、风险等级、修复建议 |
| SQL/视图生成 | 需求、关系路径、字段清单 | SQL、关联解释、风险点 |
| 探库报告生成 | 系统摘要、表画像、关系候选 | Markdown 报告 |

#### Dify 输入原则

- 一次只发一张表或一个关系候选集合。
- 不发送真实患者数据。
- 样本值必须脱敏。
- 大字段只发送摘要、长度、是否为空。
- 提示词固定版本，便于审计。

#### Dify 输出 JSON 示例

```json
{
  "business_domain": "住院",
  "business_meaning": "住院主记录表",
  "primary_key_candidates": [["PATIENT_ID", "VISIT_ID"]],
  "important_columns": [
    {"column": "PATIENT_ID", "meaning": "患者ID", "confidence": 0.95},
    {"column": "VISIT_ID", "meaning": "住院次数", "confidence": 0.90}
  ],
  "relation_candidates": [
    {
      "from": "HIS.PAT_VISIT.PATIENT_ID",
      "to": "HIS.PAT_MASTER_INDEX.PATIENT_ID",
      "confidence": 0.95,
      "reason": "字段名一致，业务语义均为患者主索引"
    }
  ],
  "risk_notes": ["DEPT_DISCHARGE_FROM 需要关联 DEPT_DICT 获取科室名称"]
}
```

## 13. 关键风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 数据库版本老旧 | 连接失败 | 支持 thick client / JDBC / ODBC 多方式 |
| 大表画像慢 | 影响生产库 | 采样、限流、夜间任务、超时控制 |
| AI 误判关系 | SQL 错误 | 关系置信度 + 人工确认 |
| Dify 输出不稳定 | 入库失败 | JSON Schema 校验 + 重试 + 人工复核 |
| 敏感数据泄露 | 安全风险 | 脱敏、权限、审计、禁止外发原文 |
| 多医院表名差异 | 复制困难 | 业务实体映射 + 同义字段配置 |
| SQL 全表扫 | 影响数据库 | SQL 沙箱 + 默认采样 + 风险提示 |
| 质控规则误报 | 增加人工工作量 | 规则分级、试运行、人工确认后启用 |
| 字段无用误判 | 误导开发人员 | 不自动删除，只标记状态并保留引用证据 |

- 可生成元数据快照和表字段变更事件。
- 可配置并执行基础质控规则。
- 可输出字段空值率、唯一率、值域异常、范围异常。

### 阶段二验收

- 可识别核心关系候选。
- Dify 可返回结构化 JSON 分析结果。
- AI 字段解释可入库。
- 关系可人工确认。

### 阶段三验收

- AI 可基于已确认关系生成 SQL。
- SQL 沙箱可拦截非 SELECT 和危险语句。
- SQL 可小样本执行并记录审计。
- AI 可推荐质控规则，规则需人工确认后启用。
- 可生成数据质量检查报告。

### 阶段四验收

- 可展示表关系图。
- 可查询两张表之间的关系路径。
- 可展示视图依赖关系。

## 15. 推荐结论

建议采用“成熟框架打底 + 医疗核心能力自研”的方式。

第一版不建议直接上大型数据目录平台，也不建议一开始强依赖 Neo4j。应先完成内网探库、元数据画像、快照 diff、Dify 分析推送和人工确认闭环。

推荐第一版技术组合：

```text
FastAPI
+ PostgreSQL
+ pgvector
+ Vue3 + Element Plus
+ AntV G6
+ Dify Workflow
+ sqlglot SQL沙箱
+ 可视化质量规则中心
```

第二版再引入：

```text
Neo4j / Qdrant / 更完善的AI Agent工具调用 / 多租户知识包
```

最终平台应形成：

```text
数据资产门户
+ 元数据中心
+ 关系图谱
+ 字典变化检测
+ AI数据助手
+ SQL/视图资产库
+ 数据质量报告中心
```

这样既能满足当前医院内网环境，也具备后续多系统扩展、多医院复制和长期维护能力。

## 16. 复核修订与落地补充

本节针对方案复核中提出的关键问题进行补充，作为后续立项、开发和实施时必须遵循的修订意见。原方案总体方向不变，但需要在外网大模型使用、元数据库选型、探库负载、SQL 沙箱、多租户、关系推断、规则冲突、Dify 幂等、模型能力边界、成本和应急等方面进一步细化。

### 16.1 外网大模型首轮探索的修订方案

医院内网环境下，外网大模型不应作为平台主链路。考虑当前可通过前置机或本机电脑连通外网，可以将外网强模型作为“首轮增强分析工具”，但主线必须支持纯内网运行。

#### 16.1.1 推荐定位

```text
主链路：内网探库服务 + 内网 Dify + 32B 模型
增强链路：前置机/本机电脑中转 + 外网强模型 + 脱敏元数据包
```

外网强模型只用于：

- 首轮元数据摘要分析。
- 初版业务实体归类。
- 初版字段含义推断。
- 初版关系候选建议。
- 初版质控规则建议。
- 复杂问题第二意见。

外网强模型不用于：

- 直接连接数据库。
- 执行 SQL。
- 接收患者明细数据。
- 接收病历、报告、检验结果原文。
- 保存医院真实连接信息。

#### 16.1.2 前置机/本机外发通道

推荐采用“脱敏包导出 + 审批 + 外网分析 + 结果回流”的方式，而不是让内网服务直接调用外网 API。

```text
内网探库服务
  -> 生成脱敏元数据包 metadata_package.json
  -> 信息科复核
  -> 前置机/本机电脑外发给外网模型
  -> 外网模型返回 analysis_result.json
  -> 人工复核
  -> 导入内网元数据库
```

脱敏元数据包允许包含：

```text
系统名、schema、表名、字段名、字段类型、字段长度、字段注释、行数、空值率、唯一率、最大长度、枚举值脱敏样本、字段语义摘要、历史 SQL 去敏版本
```

禁止包含：

```text
姓名、身份证、电话、地址、住院号原值、门诊号原值、医保号原值、病历文本原文、检查报告原文、检验结果原文、数据库 IP、端口、账号、密码
```

#### 16.1.3 合规清单

启用外网模型前，需要至少完成以下确认：

- 是否符合医院信息安全制度。
- 是否需要数据出境或外发审批。
- 是否需要与模型服务商签署数据处理协议。
- 是否保留外发内容、接收结果和审批记录。
- 是否能证明外发内容不含个人敏感信息。
- 是否设置人工复核人和责任人。

#### 16.1.4 完全禁止外网时的备选方案

如果医院禁止任何外发，则使用纯内网模式：

```text
内网探库 -> PostgreSQL 元数据库 -> Dify Workflow -> 32B 模型分析 -> 人工确认
```

纯内网模式下，首轮探索可以降低目标：先完成表画像、字段画像、关系候选和基础质控规则，不要求一次性生成完整业务图谱。

### 16.2 元数据库选型修订：PostgreSQL 与 Oracle 双方案

原方案推荐 PostgreSQL 作为平台元数据库，该方向仍然推荐，但需要承认新增 PostgreSQL 会带来运维成本。因此修订为“双方案”。

#### 16.2.1 方案 A：PostgreSQL 作为元数据库（推荐）

适合条件：

- 信息科可以维护 PostgreSQL。
- 允许新增开源数据库组件。
- 希望使用 pgvector。
- 后续考虑多医院复制部署。
- 希望规则配置、画像结果、AI JSON 入库更灵活。

需要补充的运维能力：

- 定期备份与恢复演练。
- PostgreSQL 版本升级策略。
- 数据目录、元数据和 AI 结果的容量规划。
- 监控：连接数、慢 SQL、磁盘、WAL、锁等待。
- 权限：应用账号、只读账号、备份账号分离。

#### 16.2.2 方案 B：使用现有 Oracle 作为元数据库（备选）

适合条件：

- 医院不希望新增 PostgreSQL。
- 已有 Oracle DBA 和成熟备份机制。
- 元数据规模不大。
- 第一版只做探库和关系管理，不强依赖向量检索。

替代方式：

```text
PostgreSQL JSONB -> Oracle CLOB/JSON 字段
pgvector -> Qdrant 独立向量库 或 Dify 知识库
PostgreSQL 关系边表 -> Oracle 普通关系表
```

风险：

- JSON 查询和索引维护复杂度更高。
- 后续多医院复制部署依赖各院 Oracle 版本。
- AI 结果和规则配置这类半结构化数据维护成本更高。

#### 16.2.3 最终建议

```text
MVP 建议：PostgreSQL + pgvector。
若医院短期无法维护 PostgreSQL：先用 Oracle 元数据库 + Dify 知识库/外部 Qdrant 作为过渡。
正式推广到其他医院前：优先收敛到 PostgreSQL 方案，便于标准化部署。
```

### 16.3 探库负载治理专节

探库必须避免影响生产系统。平台应默认遵循“优先探备库、最小采样、限流执行、可中断”的原则。

#### 16.3.1 数据源优先级

```text
第一优先：ODS 备份库 / 数据中心备份库 / 只读灾备库
第二优先：业务系统只读备库
第三优先：生产库只读账号，且仅做元数据级探查
```

结合当前环境，应优先使用：

```text
10.10.8.216 ODS 备份库作为主要探库对象
避免直接扫描生产 HIS/LIS/PACS 数据库
```

#### 16.3.2 表规模分级策略

| 表规模 | 探查策略 |
|---|---|
| < 1 万行 | 可全量统计空值率、唯一率、枚举值 |
| 1 万 - 100 万行 | 可分字段采样，避免全表 count distinct |
| 100 万 - 1000 万行 | 默认采样 1%-5%，只对核心字段做画像 |
| > 1000 万行 | 默认只采元数据、行数估算、时间范围；详细画像需人工批准 |

#### 16.3.3 字段画像分级

字段按重要性分级：

```text
一级字段：主键候选、患者ID、住院号、门诊号、检查号、检验单号、时间字段
二级字段：编码字段、字典字段、状态字段、金额字段
三级字段：普通文本字段
四级字段：CLOB/BLOB/LONG 字段，只统计长度和空值，不抽取原文
```

#### 16.3.4 数据库级保护

Oracle：

- 使用只读账号。
- 设置 profile 限制 CPU、session、idle time。
- 生产库建议通过 Resource Manager 限制探库账号资源。
- 大查询默认先 explain plan。
- 长任务可记录 session id，便于 DBA kill session。

PostgreSQL：

- 设置 `statement_timeout`。
- 设置 `idle_in_transaction_session_timeout`。
- 使用只读事务。
- 限制连接池最大连接数。

SQL Server：

- 使用只读账号。
- 设置 query timeout。
- 优先读取只读副本。
- 控制锁等待时间。

#### 16.3.5 探库任务熔断

满足任一条件自动中止：

- 单 SQL 执行超过阈值。
- 单数据源活跃探库连接超过阈值。
- 数据库 CPU 或等待事件超过阈值。
- 连续失败次数超过阈值。
- DBA 或管理员手动暂停数据源。

### 16.4 SQL 沙箱重写方案

SQL 沙箱必须采用多层防护，不能只依赖关键词黑名单。

#### 16.4.1 四层防护

```text
第一层：数据库账号权限
第二层：SQL 解析与语法白名单
第三层：执行计划与风险评估
第四层：执行超时、行数限制、审计与脱敏
```

#### 16.4.2 数据库账号权限

- 每个数据源单独建立只读账号。
- 只授予必要 schema 的 SELECT 权限。
- 不授予 CREATE、ALTER、DROP、INSERT、UPDATE、DELETE、EXECUTE 高危权限。
- 不允许访问系统敏感包或外部调用包。
- 生产库账号与探库账号分离。

#### 16.4.3 SQL 解析白名单

使用 `sqlglot` 或等价 SQL parser 做语法树解析：

- 仅允许 SELECT / WITH SELECT。
- 禁止 DDL、DML、MERGE、CALL、EXEC。
- 禁止多语句。
- 禁止危险函数和包。
- 禁止无条件查询超大表。
- 禁止交叉连接未加限制。

#### 16.4.4 执行计划风险评估

执行前先获取执行计划：

- Oracle：`EXPLAIN PLAN FOR ...`。
- PostgreSQL：`EXPLAIN`。
- SQL Server：`SET SHOWPLAN_XML ON` 或等效方式。

拒绝条件示例：

- 估算行数超过阈值。
- 多个大表 nested loop 无过滤。
- 出现笛卡尔积。
- 未命中任何过滤条件且表规模超过阈值。
- 预计 cost 超过阈值。

#### 16.4.5 执行限制

- Oracle 自动外层包装 `WHERE ROWNUM <= N` 或 `FETCH FIRST N ROWS ONLY`。
- PostgreSQL 自动追加 `LIMIT N`。
- SQL Server 自动追加 `TOP N`。
- 设置 SQL 超时。
- 设置最大返回字节数。
- 自动脱敏敏感字段。
- 所有执行写入审计日志。

### 16.5 多租户修订方案

MVP 阶段不做多医院功能，只预留多租户字段。多医院推广时再根据部署模式选择隔离方式。

| 模式 | 说明 | 适用场景 |
|---|---|---|
| 物理隔离 | 每家医院一套平台和数据库 | 合规要求高、医院独立部署 |
| Schema 隔离 | 一个 PostgreSQL 实例，不同医院不同 schema | 集团医院、内网统一运维 |
| 行级隔离 | 所有表带 tenant_id | SaaS 化平台，但医疗场景需谨慎 |

推荐策略：

```text
MVP：单医院单租户，表结构预留 tenant_id。
院内推广：仍建议单租户。
集团部署：优先物理隔离或 Schema 隔离。
公网 SaaS：暂不建议用于医疗明细数据场景。
```

### 16.6 关系推断伪关系过滤

关系推断必须增加排除规则，避免把所有表共有字段误判为业务关系。

默认不参与自动关系推断的字段：

```text
CREATE_TIME
UPDATE_TIME
CREATE_DATE
MODIFY_TIME
OPERATOR
CREATED_BY
UPDATED_BY
ORG_ID
HOSPITAL_ID
TENANT_ID
STATUS
FLAG
REMARK
MEMO
```

过滤规则：

- 字段名过于通用且无明确业务含义，不自动建关系。
- distinct 比例过高但无业务主键含义，不自动建关系。
- 字段值覆盖率高但字段类型为状态、标志、机构、时间，不自动建关系。
- 仅当字段名、注释、值覆盖率、历史 SQL 证据同时满足时才提升置信度。

### 16.7 字典表自动识别规则

字典表识别应结合表名、字段结构、行数、更新频率和数据形态。

候选规则：

- 表名包含 `_DICT`、`DICT`、`TYPE`、`CLASS`、`ITEM`、`CATALOG`、`CODE`。
- 行数通常小于 5 万。
- 字段数相对较少。
- 存在 CODE + NAME 或 ID + NAME 字段组合。
- 枚举值稳定，更新频率低。
- 被多个业务表通过编码字段引用。

字典识别结果也应有状态：

```text
疑似字典
AI确认字典
人工确认字典
非字典
```

### 16.8 字段无用判定例外与引用证据

字段无用判定不能只看空值率。

以下情况不判定为无用字段：

- 被任何视图、报表、接口 SQL 引用。
- 属于上报标准字段。
- 属于死亡、转科、手术、输血、分娩等低频业务字段。
- 属于法律、审计、医保追溯字段。
- 在历史任一快照中出现过有效值。

字段无用结论必须给出证据：

```text
空值率连续 N 次 = 100%
未被任何 SQL 资产引用
未被任何视图依赖
非上报字段
人工确认时间
```

### 16.9 规则冲突检测

质量规则入库前需要做冲突检查。

冲突类型：

- 同一字段既配置必填，又配置允许空。
- 同一字段存在多个互斥值域。
- 同一字段范围规则上下限冲突。
- 同一关系既要求必须存在，又允许缺失。
- 同一规则不同版本同时启用。

处理策略：

- 增加规则优先级。
- 同一字段同类规则只允许一个启用版本。
- 冲突规则进入待确认状态，不自动启用。
- 规则冲突结果写入审计日志。

### 16.10 Dify 任务去重与幂等

Dify 分析任务需要避免重复推送。

建议唯一键：

```text
tenant_id + task_type + object_type + object_id + snapshot_id + prompt_version
```

幂等策略：

- 对象画像未变化时不重新推送。
- prompt_version 未变化时复用上次结果。
- 失败任务允许重试，但 retry_count 受限。
- Dify 返回结果必须通过 JSON Schema 校验后才入库。

### 16.11 AI 工具 API 分级

AI 工具需要按风险分级授权。

| 工具等级 | 工具 | 使用角色 |
|---|---|---|
| L1 检索类 | search_tables、get_table_profile、find_relation_path | 普通工程师 |
| L2 生成类 | generate_sql、explain_sql、recommend_quality_rule | 数据工程师 |
| L3 执行类 | run_sample_sql、profile_result | 授权工程师 |
| L4 持久化类 | save_sql_artifact、confirm_relation、enable_rule | 管理员/负责人 |

### 16.12 32B 模型能力边界

内网 32B 模型可支撑第一版，但需要明确边界。

适合任务：

- 单表业务含义解释。
- 字段含义分类。
- 字典变化说明。
- 质量异常解释。
- 简短 SQL 草稿。
- 报告摘要。

不适合直接承担：

- 跨 5 张以上大表的复杂 SQL。
- 超过 200 行的长 CTE。
- 多步骤自动推理和执行。
- 自动确认关系。
- 自动启用质控规则。

处理方式：

- 复杂任务拆成小任务。
- AI 只生成草稿，不直接执行。
- 通过关系库提供明确路径，减少模型自由推理。
- 必要时对脱敏元数据使用外网强模型做第二意见。

### 16.13 实施周期与人力假设

原阶段周期需要补充人力假设。

```text
4-6 周 MVP 的前提：
- 1 名后端工程师熟悉 Python/FastAPI/数据库
- 1 名前端工程师熟悉 Vue/Element Plus
- 1 名信息科工程师负责数据库账号、业务解释、测试
- 不包含厂商协调和安全审批时间
```

如果只有 1 名工程师推进，MVP 周期建议调整为 8-12 周。

| 人力 | 2-3 人小组 | 后端、前端、信息科业务确认 |

### 16.16 术语表与报告模板

方案应补充术语表，便于管理人员阅读。

示例：

### 16.18 探库任务应急与 kill 机制

每个探库任务必须可暂停、可取消、可 kill。

平台需记录：

```text
task_id
worker_id
data_source_id
db_session_id
current_sql_hash
start_time
timeout_seconds
status
```

应急流程：

```text
1. 管理员在任务中心点击暂停数据源。
2. Worker 停止提交新 SQL。
3. 对运行中 SQL 发出取消请求。
4. 必要时由 DBA 根据 session_id kill session。
5. 系统记录应急日志。
6. 复盘该 SQL 是否加入高危规则黑名单。
```
