# 人员与权限维护功能开发执行计划

> 类别：方案
> 日期：2026-07-06
> 目标：新增人员主数据维护与平台角色权限维护能力。人员信息以 HRP 与 HIS 人员字典同步为主，平台不手工维护基础人员档案；角色、菜单、页面和操作权限由数据治理平台独立维护。

## 1. 背景与现状

当前系统已有 `identity` 模块基础表与接口：`asset_identity_persons`、`asset_identity_departments`、`asset_identity_accounts`、`asset_identity_sync_diffs`，可查询人员、科室、账号和同步差异。后端也已有简化 RBAC：`asset_roles`、`asset_user_roles`、`asset_role_permissions`，`main.py` 中按 URL 前缀做角色拦截。

现有不足：人员同步仍是占位；角色、菜单、页面、按钮权限没有维护界面；API Token 与人员绑定、用户角色授权、前端菜单过滤、后端权限校验尚未形成完整闭环。

## 2. 建设目标

1. 建立人员主数据同步：从 HRP 与 HIS 人员字典采集，生成统一人员、科室、系统账号和差异清单。
2. 建立平台权限模型：支持给某个人赋予角色，角色控制可访问的菜单、页面、接口资源和关键按钮。
3. 建立可视化维护界面：人员列表、人员画像、角色管理、人员授权、权限矩阵、API Token 绑定。
4. 建立安全闭环：前端隐藏无权限菜单，后端强制校验接口权限，权限变更写审计日志。

## 3. 权限设计原则

人员信息不在平台中直接造主档；仅允许补充平台账号绑定、角色授权、备注和复核状态。人员姓名、工号、科室、在职状态以 HRP/HIS 同步结果为准。

角色由平台独立维护，建议内置：`platform_admin`、`asset_viewer`、`asset_editor`、`quality_admin`、`identity_admin`、`dict_admin`、`ops_admin`、`approver`、`ai_user`。后续可新增业务角色，如“病案质控员”“字典维护员”。

权限粒度分三层：菜单权限控制入口显示，页面权限控制路由访问，操作权限控制按钮和后端接口。以后端权限为准，前端只负责体验。

## 4. 数据模型整改

复用现有表：

| 表 | 用途 | 整改点 |
|---|---|---|
| `asset_identity_persons` | 统一人员主档 | 增加 HRP/HIS 来源优先级、人员状态、最后同步时间展示 |
| `asset_identity_accounts` | 系统账号映射 | 增加平台账号、HIS账号、HRP账号、EMR账号等绑定展示 |
| `asset_roles` | 平台角色 | 增加启停、排序、是否系统内置、数据范围字段 |
| `asset_user_roles` | 人员角色授权 | 建议用 `person_code` 或平台账号作为 `user_identifier`，增加唯一约束和过期时间 |
| `asset_role_permissions` | 角色权限 | 扩展 `resource_type/menu/page/api/button`、`resource_code`、`action`、`effect` |
| `asset_api_keys` | API Token | 强制绑定 `user_identifier`，逐步取消未绑定 Token 放行逻辑 |

建议新增表：

| 表 | 用途 |
|---|---|
| `asset_permission_resources` | 维护系统菜单、页面、接口、按钮资源清单 |
| `asset_user_permission_overrides` | 特殊人员临时授权或禁用，默认少用 |
| `asset_permission_audit_logs` | 权限授予、撤销、角色变更审计 |

## 5. 后端接口计划

新增或完善 `/api/v1/identity`：

- `POST /identity/sync/hrp`：同步 HRP 人员、科室、账号。
- `POST /identity/sync/his`：同步 HIS 人员字典。
- `GET /identity/persons`：支持姓名、工号、科室、在职状态、来源系统筛选。
- `GET /identity/persons/{person_code}`：人员画像，展示来源、账号、角色、权限。

新增 `/api/v1/permissions`：

- `GET/PUT /roles`：角色列表、新增、编辑、启停。
- `GET /resources`：菜单、页面、接口、按钮资源树。
- `PUT /roles/{role_code}/permissions`：维护角色权限矩阵。
- `PUT /users/{user_identifier}/roles`：给人员赋权或撤权。
- `GET /users/{user_identifier}/permissions`：计算人员最终权限。
- `GET /me/permissions`：前端登录后获取菜单和按钮权限。

同步逻辑必须只读查询 HRP/HIS 源库，不对源库写入。平台授权只写本平台 PostgreSQL。

## 6. 前端页面计划

在“身份与权限”模块下新增或整改：

1. 人员管理：表格展示人员编码、姓名、科室、岗位、在职状态、来源系统、最近同步时间；支持进入人员画像。
2. 人员画像：展示 HRP/HIS 来源记录、账号绑定、当前角色、权限摘要、授权历史。
3. 角色管理：维护角色编码、中文名、说明、启停、内置标识。
4. 权限矩阵：左侧角色，右侧资源树；按菜单、页面、接口、按钮分组勾选。
5. 人员授权：选择人员后勾选角色，支持到期时间、授权原因、撤销。
6. Token 绑定：API Token 绑定到人员，避免旧 Token 绕过角色控制。

界面展示尽量中文化，权限资源名称使用“资产总览、数据源管理、关系复核、质量规则、诊断手术维护、运维工具”等业务名称。

## 7. 鉴权整改

后端当前未绑定 Token 会兼容放行。整改分两步：

1. 过渡期：未绑定 Token 仅允许访问只读资产页面，写操作和管理接口必须绑定人员。
2. 正式期：所有受保护接口必须绑定人员，并按 `asset_role_permissions` 校验。

前端路由使用 `/me/permissions` 返回结果动态过滤菜单；按钮用权限码控制显示，例如 `quality.rule.create`、`dict.medical.disable`、`identity.role.grant`。

## 8. 开发顺序

1. 后端模型与 Alembic：补资源、审计、权限扩展表，保留现有表不删。
2. 权限种子数据：初始化菜单/页面/API/按钮资源和内置角色。
3. 人员同步执行器：实现 HRP/HIS 只读同步，生成差异记录。
4. 权限 API：角色、资源、人员授权、我的权限。
5. 后端中间件：从硬编码 URL 前缀升级为资源权限校验。
6. 前端页面：人员管理、人员画像、角色管理、权限矩阵、人员授权。
7. 验收与审计：补测试、权限拒绝用例、审计日志核查。

## 9. 验收标准

- HRP/HIS 人员同步后，`asset_identity_persons`、`asset_identity_accounts` 有数据且可追溯来源。
- 可在前端给某个人授予“质量管理员”角色，该人员只能看到并操作质量规则相关页面。
- 无权限人员访问受保护接口返回 403，前端菜单不显示无权限入口。
- API Token 必须可绑定人员，并能通过人员角色计算权限。
- 权限新增、撤销、角色变更均写入审计日志。
- 后端通过 `pytest tests/ -q`，前端通过 `pnpm run typecheck && pnpm run build`。

## 10. 待确认问题

1. HRP 人员源表、字段、连接方式是否已经确定；若未确定，先以 HIS 人员字典做第一版同步。
2. 平台登录最终使用 API Token、本地账号，还是对接医院统一认证。
3. 是否需要按科室、系统、数据域做数据范围权限，例如只能看本科室资产或只看 HIS。
4. 权限变更是否需要审批流；若需要，复用 `asset_govern_change_requests`。
5. 医院是否已有标准角色清单；若没有，先按本计划内置角色落地。
