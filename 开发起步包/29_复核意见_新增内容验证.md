### 2.9 `check_sql` 模板参数化方案缺失

29 号计划的 `check_sql` 字段是纯文本 SQL，但质控规则需要**跨表复用**。例如"主键唯一性校验"规则应用到 100 张表时，SQL 中需要 `:table_name`、`:pk_column` 等占位符。29 号计划未说明参数化方案。

**建议**：`check_sql` 需要定义参数占位符规范：
- 使用 `:table_name`/`:column_name`/`:schema_name` 命名占位符（Oracle 和 PostgreSQL 均支持 `:param` 绑定语法）
- 执行时由 `quality_sql_runner.py` 从规则的 `target_table/target_field` 字段读取参数值替换
- SQL 中禁止拼接用户输入，防止 SQL 注入
- 示例：`SELECT COUNT(*) FROM :schema_name.:table_name WHERE :pk_column IS NULL` → 引擎替换为 `SELECT COUNT(*) FROM HIS.PAT_VISIT WHERE PATIENT_ID IS NULL`

### 2.15 规则执行性能约束缺失

如果同时启用 50+ 条 SQL 规则对大表执行质控，可能产生严重性能问题。29 号计划未说明执行策略。

**建议**：
- 单条规则 SQL **超时 30 秒**，超时后标记为 `rule_error`
- 整批规则**串行执行**（源库只读不能并行施压）
- 总执行超时 **10 分钟**，超时后 check_run 标记为 `timeout`
- 大表（`HIS.LAB_RESULT`/`INP_BILL_DETAIL`/`ORDERS`）的规则必须带 WHERE 条件或子查询限定
- `quality_sql_runner.py` 中设置 `statement_timeout`（PostgreSQL）或 `CALL_TIMEOUT`（Oracle）
