# 应用层选型与开发方案

> 基于 `17_内网部署环境与服务器信息.md` 与 `18_基础设施部署方案.md` 的后续应用层方案。
> 当前基础设施已完成：8.83 上 `data_asset` 元数据库、资产导入（865 表 / 26894 字段 / 47 关系）、pgvector 0.7.4、备份脚本；应用代码尚未启动。

---

## 1. 结论

推荐采用：

```text
后端核心能力：自研 FastAPI
前端管理端：基于成熟 Vue3 Admin 模板二次开发
图谱/表格/SQL编辑器：选用成熟前端组件库
外网 AI 协作：只暴露脱敏元数据与关系工具 API，不暴露业务库和患者数据
```

不建议直接拿完整开源数据目录平台（如 DataHub / OpenMetadata / Amundsen）二开作为主平台。

原因：医院本项目的关键能力不是通用数据目录，而是**已确认医疗关系图谱 + Oracle 旧库探查 + SQL 沙箱 + Dify/外网 AI 受控工具调用 + 视图生成辅助**。这些能力高度定制，套大平台会带来部署复杂、模型不匹配和安全边界难控的问题。

---

## 2. 开源二开 vs 自研对比

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 完整开源数据目录平台二开 | 功能多，已有元数据/搜索/血缘框架 | 依赖重，常见需要 Kafka/Elastic/K8s/Java 服务；内网离线部署复杂；医疗关系/SQL沙箱/Dify工具调用仍要重写 | 不推荐作为主平台 |
| 纯从零自研前后端 | 最贴合业务，架构可控 | 前端后台基础设施、权限、布局、表格等会重复造轮子 | 后端适合自研，前端不建议从零 |
| **自研 FastAPI + Vue Admin 模板二开** | 核心能力自控；前端节省 40%-60% 基础工作；离线交付可控 | 需要自己设计数据模型、API、SQL沙箱和AI工具 | **推荐** |

## 3. 推荐技术组合

### 3.1 后端

| 模块 | 推荐 | 说明 |
|---|---|---|
| API 框架 | FastAPI | 与 `01` 架构一致，适合工具 API / 异步任务 / 文档化接口 |
| ORM | SQLAlchemy 2.x | 连接 8.83 `data_asset` |
| 配置 | pydantic-settings + `.env` | 密码不入库，不进 git |
| PG 驱动 | psycopg | 访问 `data_asset` |
| Oracle 驱动 | oracledb thick | 访问 8.216/后续 HIS；复用 8.53 已有 instantclient 包 |
| SQL 安全 | sqlglot + 自研规则 | SELECT 白名单、多语句拦截、LIMIT/ROWNUM 包装 |
| 后台任务 | APScheduler 起步 | MVP 足够；后续需要并发再上 Celery/RQ |
| AI/Dify 调用 | httpx | 调 8.53 Dify API；外网 AI 只调工具 API |

### 3.2 前端

| 模块 | 推荐 | 说明 |
|---|---|---|
| 框架 | Vue3 + TypeScript | 与 `01` 一致 |
| UI | Element Plus | 医院信息科维护成本低 |
| 后台模板 | Geeker-Admin 或 vue-pure-admin | 只取布局、路由、菜单、权限、表格基础，不照搬业务 |
| 图谱 | AntV G6 | 表关系图、字段链路图 |
| 图表 | ECharts | 总览、趋势、质量统计 |
| SQL 编辑器 | Monaco Editor | SQL 沙箱与视图草稿体验 |

## 4. GitHub 使用方式

当前网络条件：

```text
本机 Windows：可访问 GitHub / npm / pypi
8.53 跳板机：无外网
8.83 应用服务器：无外网
```

因此开发交付方式应固定为：

```text
本机拉 GitHub / npm / pypi
  -> 本机构建前端 dist、下载 Python wheels、打包后端 release
  -> scp 到 8.53:/tmp
  -> askpass/scp 到 8.83:/home/deploy/app
  -> 8.83 离线安装 / systemd 启动
```

建议不要在 8.83 上 `git clone`、`pip install`、`npm install`，因为服务器无外网，会不可重复。

## 5. 外网 AI 如何安全分析表关系与写视图

外网 AI **不能直连内网数据库**，也不能拿真实患者数据。推荐两种模式。

### 5.1 安全优先：脱敏包导出模式（推荐）

平台提供导出接口：

```text
export_ai_context(task)
```

导出内容仅包括：

```text
表名、字段名、字段类型、字段注释、已确认关系、join 条件、行数级别、验证状态、视图 DDL 去敏片段
```

不导出：

```text
患者姓名、身份证、电话、地址、病历文本、报告原文、检验结果明细、账号密码、真实样本值
```

流程：

```text
FastAPI 生成脱敏 JSON/Markdown 上下文
  -> 人工复核
  -> 提交给外网 AI
  -> AI 返回 SQL/视图草稿
  -> 导入平台 SQL 沙箱校验
  -> 人工确认后使用
```

### 5.2 受控工具 API 模式（后续可选）

FastAPI 暴露只读工具 API，供 AI 通过受控通道调用：

```text
search_tables(keyword)
get_table_profile(schema, table)
get_confirmed_relations(table)
find_relation_path(from_table, to_table)
get_view_seed(view_name)
generate_sql_context(task)
validate_sql(sql)
```

边界：

- 只读元数据，不返回患者明细。
- 默认不提供 `run_sql` 给外网 AI。
- 如必须执行 SQL，只走内网人工触发的 SQL 沙箱。
- 所有请求带 token、审计日志、限流、结果脱敏。

## 6. MVP 应用层范围

### 6.1 第一阶段：只读资产门户 + AI 工具 API

优先实现最小可用闭环：

| 页面/API | 内容 |
|---|---|
| 数据资产总览 | 表数、字段数、关系数、验证状态 |
| 表清单 | 查 schema/table/domain/source |
| 表详情 | 字段、主键候选、备注、关系 |
| 关系查询 | 某表上下游、两表路径 |
| AI 上下文导出 | 按任务导出脱敏元数据包 |
| AI 工具 API | `search_tables` / `get_table_profile` / `find_relation_path` / `get_confirmed_relations` |

这一阶段不直接执行业务 SQL，风险最低。

### 6.2 第二阶段：SQL 沙箱 + 视图草稿

新增：

- SQL 解析白名单（SELECT only）。
- Oracle 11g 兼容 ROWNUM 限制。
- 大表风险提示（`LAB_RESULT`、`INP_BILL_DETAIL`、`ORDERS`）。
- 视图草稿保存、版本、人工确认。

### 6.3 第三阶段：探库 Worker 与变更检测

新增：

- 8.216 ODS 定时元数据刷新。
- 快照 diff。
- Dify 变化影响分析。

## 8. 推荐执行路线

1. 在本机新建代码仓库 `data-asset-platform`。
2. 后端先自研 FastAPI 最小骨架，直接读取 8.83 `data_asset.asset_*` 表。
3. 前端从 Geeker-Admin 或 vue-pure-admin 拉模板，只保留布局/登录/菜单/表格基础。
4. 本机完成首次构建与离线包准备。
5. 通过 8.53 中转到 8.83 `/home/deploy/app/`。
6. 8.83 用 `systemd + venv` 启动后端；前端 dist 后续由 Nginx 托管。
7. 先上线只读资产门户与 AI 上下文导出，再做 SQL 沙箱。

## 9. 最终建议

**不要找一个完整开源数据治理平台直接改。**

更稳的路线是：

```text
业务内核自研：FastAPI + data_asset + SQL沙箱 + AI工具API
前端工程复用：Vue3 Admin 模板 + Element Plus + G6 + Monaco
部署方式固定：本机联网构建/打包 -> 跳板机 -> 8.83 离线 systemd 运行
```

这样既能利用 GitHub 开源生态，又不会被重型平台绑架，也能满足医院内网、安全、Oracle 旧版和关系图谱的特殊要求。
