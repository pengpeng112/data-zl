# 平台能力深化与 AI 协作路线图

> 本文档基于当前代码库（`F:\python\data-asset-platform`）实测状态和既有文档（`01/09/10/24` 等）编写，目标是把平台从"只读资产门户"升级为：
> 1. **表关联关系分析平台**
> 2. **数据质量控制与问题发现平台**
> 3. **AI 可持续探索/查询/写视图的协作平台**
>
> 不局限于以上三项，本文档同时列出治理、血缘、协作、运维等方向的建议。
> 编写：2026-07-04。

> 核查补充（2026-07-04）：本文已按当前代码库与 `11_视图关系解析与HIS分类报告.md` 复核。原判断总体成立，但需精确区分：`772` 是 ODS 视图表依赖数，`732` 才是静态解析出的疑似 join 边；二者都尚未进入应用层。下文已按这一口径修订。

## 0. 现状盘点（代码实测，不是猜测）

### 0.1 后端现状

```
backend/app/
├── api/v1/
│   ├── ai.py          # 只有 export-context（脱敏元数据导出），无 AI 交互能力
│   ├── graph.py        # 全局图谱 / 邻居图 / 筛选项，3 个接口
│   ├── health.py        # 健康检查
│   ├── relations.py    # 两表最短路径 BFS
│   └── tables.py        # summary / 表清单 / 表详情 / 字段搜索 / sort
├── core/                # config / db / exceptions / logging
├── models/asset.py      # 仅 3 张表：asset_tables / asset_columns / asset_relations
└── schemas/             # asset / common / graph
```

- 数据库只有 **3 张资产元数据表**，没有质量规则表、质量结果表、AI 会话表、视图草稿表、审计表、变更快照表。
- 关系数据来自静态 CSV 导入的 **47 条正式关系**；另有 **772 条 ODS 视图表依赖**和 **732 条疑似 join 边**躺在 `数据资产_关系图谱/` 里，尚未进入应用层。
- 无任何数据质量检查逻辑，无任何调度任务（无 Celery/APScheduler/cron）。
- 无 AI 交互式工具 API，`export-context` 只是"一次性打包给外部 AI 看"，不支持 AI 多轮探索、不支持 AI 提议 SQL、不支持视图落地。
- 无鉴权、无用户体系、无操作审计。
- 无 SQL 执行能力（P2 沙箱未开发）。

### 0.2 前端现状

```
frontend/src/views/asset/
├── overview        # 总览：4 卡片 + 4 图表
├── tables           # 表清单
├── table-detail     # 表详情 + 邻居图 + 字段搜索
├── graph             # 全局关系图谱
├── relations         # 关系路径查询（图形化）
├── ai-context        # AI 上下文导出（选表→导出 JSON）
└── components/RelationGraph.vue  # 通用 ECharts 图谱组件
```

- 5 个页面，功能都停留在"查看/查询"层面，没有"分析/诊断/协作"层面的能力。
- 无质量问题页面、无 AI 对话式界面、无视图草稿管理界面。

### 0.3 数据/知识现状（已有但未进应用层的资产，非常宝贵）

| 资产 | 位置 | 当前状态 |
|---|---|---|
| 47 条正式关系 + 验证指标（孤儿率等） | `数据资产_资产包/relationships.csv` | 已进应用层 |
| 772 条 ODS 视图表依赖（从 119 个视图 DDL 解析） | `数据资产_关系图谱/ods_view_dependencies.csv` | **未进应用层**，适合做视图血缘和表使用频次 |
| 732 条疑似 join 边（从视图 join 条件解析） | `数据资产_关系图谱/ods_view_join_edges.csv` / `candidate_edges_ranked.json` | **未进应用层**，适合做候选关系工作台 |
| 25 个专题验证文档（`10~23`） | `开发起步包/` | 已知的数据坑、验证结论，**未结构化进数据库**，无法被 API 消费 |
| 已知数据质量问题（AGENTS.md 明确列出） | 文档 | **未做成规则、未做成检查任务** |
| HIS 源端资产包（1234 表/19831 字段/33 关系） | `数据资产_HIS源端资产包/` | **未导入应用层**，当前只有 865 表（含跨库），源端全貌未接入 |

**关键判断**：不缺"分析原料"，缺"把原料变成产品"的应用层能力。ODS 视图依赖、候选 join 边、专题验证文档、HIS 源端资产包目前都主要躺在文件系统里，没有变成可查询、可检查、可被 AI 调用的服务。这是本轮改进的最大杠杆点。

## 1. 三大核心能力：现状 → 目标 → 差距

### 1.1 能力一：表关联关系分析

**现状**：能查 47 条正式关系的路径和邻居图，仅此而已。

**目标**：让信息科/开发者/AI 能回答这些问题：
- 这张表还有哪些"疑似关联"没确认？（732 条疑似 join 边）
- 这张表是不是"核心表"？（度数、被引用次数）
- 如果这张表字段变了，会影响哪些视图/表？（影响分析）
- 两张表除了最短路径，还有没有其他路径？（多路径枚举）
- 某个业务域（如"检验"）内部的表都怎么连起来的？（子图查看）
- 关系的可信度是怎么来的？（点开就能看到验证过程和 SQL）

**差距**：
| 差距 | 说明 |
|---|---|
| 候选关系未接入 | 732 条疑似 join 边只在静态文件，API 查不到，图谱看不到 |
| 无影响分析 | 改一个字段/表，不知道影响谁 |
| 无多路径 | BFS 只返回一条最短路径，实际业务里可能有多条合理路径 |
| 无子图/域视图 | 无法只看"检验域"或"手术域"的关系子图 |
| 无关系解释可回溯 | 点开一条关系，看不到当初怎么验证的（哪份验证报告、什么 SQL） |
| 源端资产未接入 | HIS 源端 1234 张表/33 条关系是独立资产包，和当前 865 表体系是两套数据，没打通 |

### 1.2 能力二：数据质量控制与问题发现

**现状**：**完全没有**。没有规则、没有检查、没有问题列表、没有质量分数。

**目标**：
- 系统能自动/半自动发现："这张表有问题"、"这条关系孤儿率超标"、"这个字段全是空值"
- 已知的历史坑（`EMR_ANES_AFTER_RECORD` ID 重复 30%、`EMR_OUTPATIENT_RECIPE_ITEM.OR_ID` 100% 孤儿、`EMR_OUTPATIENT_DIAG` 0 行等）要变成"系统里挂着的问题"，而不是"文档里写着的坑"
- 有一个"数据质量问题清单"页面，可以按严重程度、状态（待处理/已确认/已解决）跟踪
- 关系验证指标（孤儿率）已经有数据基础（`relationships.csv` 的 `validation_metrics`），应该被结构化、可查询、可展示为质量看板

**差距**：
| 差距 | 说明 |
|---|---|
| 无质量规则模型 | 没有"规则"这个概念（完整性/唯一性/孤儿率/新鲜度/波动异常） |
| 无检查执行器 | 没有任何调度或手动触发检查的机制 |
| 无问题台账 | 没有"问题"这个实体（严重程度、状态、责任人、发现时间） |
| 已知坑未结构化 | AGENTS.md 里写的坑，全靠人记，系统不知道 |
| 无质量看板 | 前端没有"数据质量"入口 |
| 无趋势 | 即使做了检查，也没有"上次检查 vs 这次检查"的对比 |

### 1.3 能力三：AI 持续探索/查询/写视图

**现状**：只有 `export-context`——AI 提前"打包快照"，不能持续交互。

**目标**（这是用户最核心的诉求，需要重点设计）：
- AI（通过 MCP 或 HTTP 工具调用）能**多轮持续**调用 API：先搜表 → 看字段 → 查关系 → 查路径 → 提议 SQL/视图 → （人工确认后）执行 → 看结果 → 继续下一轮
- AI 写出的 SQL/视图必须先落草稿、经沙箱校验、人工确认，才允许真正执行或落库
- 每一次 AI 探索/生成都要留痕（审计），方便复盘和纠错
- 支持"给我 3 个业务问题，AI 自动摸索表结构并给出候选 SQL"这种半自动分析场景

**差距**：
| 差距 | 说明 |
|---|---|
| 无工具化 API | 现有 API 是"人类点击触发"，不是"AI 可编排调用"的工具集 |
| 无 MCP/Function-calling 封装 | AI 客户端（如 Claude/GPT）无法直接把这些接口当"工具"调用 |
| 无 SQL 沙箱 | P2 计划里的 SQL 沙箱完全没开发，AI 生成的 SQL 无法被安全验证/执行 |
| 无草稿/版本管理 | AI 生成的 SQL/视图没有地方存、没有版本、没有确认流程 |
| 无会话记录 | AI 多轮探索的过程不可追溯 |
| 无反馈闭环 | 不知道 AI 生成的东西有没有被采纳 |

## 2. 不局限于以上三项：建议同时建设的能力

| 方向 | 说明 | 优先级 |
|---|---|---|
| **候选关系工作台** | 让人工/AI 快速审核 732 条疑似 join 边，提升为正式关系或标记拒绝 | 高（是关系分析能力的地基） |
| **数据血缘（表级/字段级）** | ODS 视图 ← HIS 源表 ← 字段级映射，回答"这个字段的数据从哪来" | 高 |
| **元数据变更检测** | 定期/手动对比快照，发现新增字段/表/视图变化 | 中 |
| **业务术语与数据字典** | 维护"住院号=INP_NO"这类业务黑话到字段的映射 | 中 |
| **表 Owner/责任人管理** | 谁负责这张表，出问题找谁 | 中 |
| **权限与登录** | 目前无鉴权，任何人访问 API 都能看到全部资产 | 高（安全） |
| **操作审计** | 谁在什么时间查了什么表/生成了什么 SQL | 高（伴随 SQL 沙箱） |
| **搜索体验增强** | 全文检索、拼音搜索、常用表收藏 | 低 |
| **移动端/大屏** | 给院领导看的资产概览大屏 | 低 |
| **导出报表** | 质量问题周报、关系图谱 PDF 导出 | 低 |
| **通知与告警** | 质量问题达到阈值时通知责任人 | 中（依赖质量模块先建成） |
| **HIS 源端资产包接入** | 独立的 1234 表资产包目前没有并入应用层，需要决定是合并还是独立视图 | 高 |

### 2.1 核查后补充：必须先补的"底座能力"

| 方向 | 说明 | 优先级 |
|---|---|---|
| **资产来源分层** | 当前 865 表资产包、HIS 源端 1234 表资产包、候选关系/视图血缘是不同来源，必须在模型中标明 `asset_scope/source_system/source_file`，避免混成一张图后语义错乱 | 高 |
| **关系类型分层** | 至少区分 `verified_relation`、`candidate_join`、`view_dependency`、`lineage_dependency`，不能把 772 条依赖当正式关系 | 高 |
| **质量规则执行等级** | 先做不扫源库的离线规则（基于已有 metrics/元数据），再做低成本源库抽样，最后才考虑高成本验证 | 高 |
| **任务状态机** | 候选关系审核、质量问题处理、AI 视图草稿都需要 `draft/reviewing/approved/rejected` 等状态流转，否则后期无法审计 | 中 |

## 3. 总体架构演进建议

### 3.1 保持不变的部分（避免过度设计）

- 后端仍用 FastAPI + SQLAlchemy，不引入微服务拆分。
- 数据库仍用现有 `data_asset` PostgreSQL，不引入额外存储（除非做字段语义检索才考虑 pgvector，已具备扩展条件）。
- 前端仍用 pure-admin-thin + Element Plus + ECharts，不引入新的图谱库（AntV G6 等），除非力导向图在候选关系接入后（可能上千条边）性能不够用。

### 3.2 需要新增的组件

| 组件 | 用途 | 是否必须 |
|---|---|---|
| **调度器**（APScheduler 或简单 cron + FastAPI 后台任务） | 定时跑质量检查、定时生成快照 | 质量模块必须 |
| **只读执行账号 + 独立连接池** | SQL 沙箱执行用，与业务查询连接池隔离 | AI 协作模块必须 |
| **SQL 解析器**（sqlglot） | 校验 AI/用户提交的 SQL 是否只读、是否命中大表黑名单 | AI 协作模块必须 |
| **MCP Server 封装**（可选，推荐） | 把只读工具 API 封装成 MCP 工具，方便 AI 客户端（Claude/GPT/Opencode）直接调用 | 强烈建议，符合当前 AI 生态趋势 |
| **审计日志表 + 简单 UI** | 记录 SQL 执行、AI 调用、质量问题处理 | 强烈建议 |
| **鉴权中间件**（先简单后完善，如固定内网 Token 或简单账号密码） | P1 阶段暴露的"无鉴权"风险必须收敛 | 高优先级 |

## 4. 数据模型新增设计（建议 DDL 草案）

> 以下模型均建在 `data_asset` 应用库的 `asset` schema 下，不触碰源库。命名与现有 `asset_tables/asset_columns/asset_relations` 保持一致风格。

### 4.1 候选关系表（支撑能力一）

```sql
CREATE TABLE asset.asset_candidate_relations (
    id BIGSERIAL PRIMARY KEY,
    domain TEXT,
    from_table TEXT,
    from_columns TEXT,
    to_table TEXT,
    to_columns TEXT,
    join_condition TEXT,
    source_view TEXT,          -- 来自哪个 ODS 视图 DDL 解析出来的
    source_file TEXT,          -- 如 ods_view_join_edges.csv / candidate_edges_ranked.json
    relation_type TEXT DEFAULT 'candidate_join', -- candidate_join/view_dependency/manual_suggestion
    confidence TEXT,           -- 候选关系置信度（低于正式关系）
    status TEXT DEFAULT 'candidate',  -- candidate/promoted/rejected
    reviewed_by TEXT,
    reviewed_at TIMESTAMPTZ,
    note TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

> 修订说明：`ods_view_dependencies.csv` 的 772 条记录不应直接进入 `asset_candidate_relations` 作为 join 候选；建议单独进入 `asset_view_dependencies` 或以 `relation_type=view_dependency` 且无 `join_condition` 的形式进入血缘表。真正用于候选关系审核的首批数据，应优先来自 `ods_view_join_edges.csv` 的 732 条疑似 join 边及其排名文件。

建议补充血缘依赖表：

```sql
CREATE TABLE asset.asset_view_dependencies (
    id BIGSERIAL PRIMARY KEY,
    view_name TEXT NOT NULL,
    referenced_table TEXT NOT NULL,
    referenced_schema TEXT,
    referenced_owner TEXT,
    source_file TEXT,
    dependency_type TEXT DEFAULT 'view_table',
    created_at TIMESTAMPTZ DEFAULT now()
);
```

### 4.2 数据质量规则与问题表（支撑能力二）

```sql
CREATE TABLE asset.quality_rules (
    id BIGSERIAL PRIMARY KEY,
    rule_code TEXT UNIQUE,          -- 如 REL_ORPHAN_RATE / COL_NULL_RATE / TABLE_ROW_COUNT_DROP
    rule_type TEXT,                 -- completeness/uniqueness/orphan/freshness/volume/custom
    target_type TEXT,               -- table/column/relation
    execution_mode TEXT DEFAULT 'metadata_only', -- metadata_only/imported_metric/sample_query/full_query
    description TEXT,
    threshold_config JSONB,         -- 例如 {"max_orphan_rate": 0.01}
    enabled BOOLEAN DEFAULT true,
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE asset.quality_findings (
    id BIGSERIAL PRIMARY KEY,
    rule_code TEXT REFERENCES asset.quality_rules(rule_code),
    target_type TEXT,               -- table/column/relation
    target_ref TEXT,                -- 如 HIS.PAT_VISIT 或 rel_id=12
    severity TEXT,                  -- critical/major/minor/info
    status TEXT DEFAULT 'open',     -- open/acknowledged/resolved/ignored
    metric_value TEXT,              -- 实测值，如 orphan_rate=32%
    detail JSONB,
    found_at TIMESTAMPTZ DEFAULT now(),
    resolved_at TIMESTAMPTZ,
    resolved_by TEXT,
    note TEXT
);

CREATE TABLE asset.quality_check_runs (
    id BIGSERIAL PRIMARY KEY,
    started_at TIMESTAMPTZ DEFAULT now(),
    finished_at TIMESTAMPTZ,
    triggered_by TEXT,               -- manual/schedule
    total_rules INT,
    total_findings INT,
    status TEXT                       -- running/success/failed
);
```

> 已知历史坑（`EMR_ANES_AFTER_RECORD` 重复、`EMR_OUTPATIENT_RECIPE_ITEM.OR_ID` 孤儿、`EMR_OUTPATIENT_DIAG` 空表等）应在首次质量检查上线时，作为**种子数据**直接写入 `quality_findings`（status=acknowledged，因为已知），后续新发现的问题才是 status=open。

质量规则上线顺序建议：

1. **metadata_only**：只用 `asset_tables/asset_columns/asset_relations` 和导入的 `validation_metrics`，不访问源库。例如关系孤儿率阈值、字段注释缺失、表未归域、候选关系未审核。
2. **imported_metric**：只消费既有验证结果 JSON/CSV，例如 `10/12/14/15/21` 已计算出的 orphan_rate、match_rate。
3. **sample_query**：经跳板机对源库做小样本、索引键限定查询，必须遵守 `ROWNUM <= N` 与大表保护。
4. **full_query**：仅夜间、白名单、人工批准；`HIS.LAB_RESULT` 等大表默认禁止。

### 4.3 AI 协作相关表（支撑能力三）

```sql
CREATE TABLE asset.ai_sessions (
    id BIGSERIAL PRIMARY KEY,
    session_key TEXT UNIQUE,      -- AI 客户端生成的会话标识
    purpose TEXT,                 -- 本次探索的业务目的（可选）
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE asset.ai_tool_calls (
    id BIGSERIAL PRIMARY KEY,
    session_key TEXT,
    tool_name TEXT,                -- search_tables / get_table_schema / get_relations / propose_sql ...
    request JSONB,
    response_summary TEXT,          -- 不存全量结果，只存摘要，避免表爆炸
    called_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE asset.view_drafts (
    id BIGSERIAL PRIMARY KEY,
    session_key TEXT,
    title TEXT,
    sql_text TEXT,
    purpose TEXT,
    status TEXT DEFAULT 'draft',   -- draft/reviewing/approved/rejected/executed
    risk_flags JSONB,               -- SQL 沙箱扫描结果（是否命中大表、是否只读等）
    created_at TIMESTAMPTZ DEFAULT now(),
    reviewed_by TEXT,
    reviewed_at TIMESTAMPTZ,
    feedback TEXT                     -- 采纳/拒绝/修改原因
);

CREATE TABLE asset.sql_audit_log (
    id BIGSERIAL PRIMARY KEY,
    executed_by TEXT,
    sql_text TEXT,
    row_count INT,
    duration_ms INT,
    status TEXT,                     -- success/blocked/error
    block_reason TEXT,
    executed_at TIMESTAMPTZ DEFAULT now()
);
```

> 边界修订：AI 协作不等于让外部 AI 直连数据库。AI 只能调用元数据/关系/血缘工具，`propose_sql` 只保存草稿并做静态风险扫描。真正执行 SQL 必须由内网人工触发，且执行位置建议优先放在**应用元数据库或只读副本/沙箱**；如确需访问源库，必须另行确认只读账号、白名单表、超时、行数限制和大表禁扫策略。

### 4.4 元数据变更快照（支撑变更检测，P3 已规划，此处细化）

```sql
CREATE TABLE asset.metadata_snapshots (
    id BIGSERIAL PRIMARY KEY,
    snapshot_time TIMESTAMPTZ DEFAULT now(),
    scope TEXT,                       -- 采集范围，如 schema=HIS
    table_count INT,
    column_count INT,
    raw_ref TEXT                      -- 指向快照文件或 JSON blob 位置
);

CREATE TABLE asset.metadata_change_events (
    id BIGSERIAL PRIMARY KEY,
    change_type TEXT,                -- table_added/table_removed/column_added/column_type_changed/view_ddl_changed
    target_ref TEXT,
    before_value JSONB,
    after_value JSONB,
    detected_at TIMESTAMPTZ DEFAULT now(),
    reviewed BOOLEAN DEFAULT false
);
```

> 全部通过 Alembic 迁移新增，不允许手工建表；每次新增表必须补迁移脚本和回滚说明（按 `24` 文档纪律执行）。

### 5.3 能力三新增 API（重点设计——AI 工具化接口）

> 设计原则：**接口即工具**，每个接口对应一个 AI 可调用的"函数"。全部只读、有严格入参校验、有速率限制建议。

| 工具名 | 接口 | 说明 |
|---|---|---|
| `search_tables` | `GET /api/v1/ai/tools/search-tables?keyword=&domain=` | 按关键词/业务域找表 |
| `get_table_schema` | `GET /api/v1/ai/tools/table-schema?table=` | 拿字段清单、类型、注释 |
| `get_relations` | `GET /api/v1/ai/tools/relations?table=` | 拿某表的关联关系 |
| `get_path` | `GET /api/v1/ai/tools/path?from=&to=` | 拿两表路径 |
| `search_columns` | `GET /api/v1/ai/tools/search-columns?keyword=` | 按业务词找字段（如"住院号"） |
| `propose_sql` | `POST /api/v1/ai/tools/propose-sql` | AI 提交生成的 SQL/视图草稿，进沙箱静态校验，返回风险提示，写入 `view_drafts`（status=draft，不执行） |
| `list_drafts` | `GET /api/v1/ai/tools/drafts?session_key=` | 查看本会话已生成的草稿 |
| `execute_sql`（人工触发，不对 AI 开放） | `POST /api/v1/sql/execute` | 内网人工确认后才能执行，走沙箱四层防护 |

**MCP 封装建议**：把以上只读工具（除 execute_sql）封装为一个 MCP Server（Python `mcp` SDK 或 FastMCP），这样 Claude/GPT/Opencode 等 AI 客户端可以直接把这些当"工具"调用，无需再手写 HTTP 客户端代码。这是让"AI 持续探索"落地的**最推荐路径**，比自建对话界面更轻量、更符合当前 AI 生态标准。

## 7. 安全与治理红线（贯穿所有阶段）

1. 源库 8.216/HIS 全程只读，任何"执行"能力都只作用于沙箱/只读账号，且只对内网人工开放。
2. AI 工具 API 一律只读，不暴露任何写接口给 AI；`propose_sql` 只是"生成建议"，不等于"执行"。
3. `execute_sql` 只对内网人工开放，且必须走 sqlglot 校验 + 只读事务 + 超时 + 行数限制 + 审计。
4. 患者相关表、字段一律不导出明细数据，AI 上下文/质量问题/审计日志中不得出现真实业务数据样本。
5. 鉴权补齐前，暂不对外网开放本平台任何写接口。
6. 质量检查涉及扫描源库时，必须遵守 `AGENTS.md` 的"大表只采元数据、禁止全表扫描、夜间执行"等既有硬约束。

## 8. 风险与应对

| 风险 | 应对 |
|---|---|
| 732 条疑似 join 边质量参差不齐，772 条依赖又不是 join 关系，直接展示可能误导 | 候选 join、视图依赖、正式关系必须分层建模并用不同样式展示；默认只展示正式关系 |
| 质量检查规则设计不当，产生大量噪音问题 | 首批规则只覆盖已确认坑 + 关系孤儿率，逐步扩展，避免"狼来了" |
| AI 工具 API 被滥用（高频调用打爆数据库） | 加简单限流（如每会话每分钟调用上限），日志监控 |
| SQL 沙箱设计不足导致误跑危险语句 | 沙箱必须四层防护（权限隔离+SQL解析+执行保护+审计），任何一层不完整都不能上线执行能力 |
| MCP 封装增加新的技术栈学习成本 | 可先用最简单的 HTTP 工具 API 起步，MCP 封装作为增强项，不阻塞核心进度 |
| 数据模型不断新增表，缺少统一治理 | 所有新表必须走 Alembic 迁移，`24`/新文档同步记录 DDL 和回滚说明 |
| HIS 源端资产包与当前 ODS 资产包范围不同，直接合并可能造成同名表歧义 | 引入 `asset_scope` 或独立 `source_asset_*` 表，先做并列展示，再决定是否融合 |
| 质量检查误扫大表影响生产库 | 默认只跑 metadata/imported_metric 规则；任何 sample/full 查询必须走白名单、超时、夜间窗口和人工批准 |
