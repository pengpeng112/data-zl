## 1. 用户新要求转译

你提出的核心问题不是“再加几个页面”，而是平台的信息架构和能力边界需要重做一层底座。当前已有资产门户、图谱、质量、AI 工具雏形，但粒度和口径仍偏散，需要改成以下主线：

| 编号 | 用户要求 | 平台整改目标 |
|---|---|---|
| R1 | 能够对院内各个数据库进行检测和质控 | 建立“数据源/系统/库/表/字段”分层资产模型，对每个数据源运行元数据检查、关系检查、质量规则、采集任务状态监控 |
| R2 | 区分系统：数据中心、HIS、移动护理等；系统下分库，再到表和字段 | 重构资产层级为 `系统 -> 数据源/库 -> schema/owner -> 表 -> 字段`，前端以树形导航和面包屑展示 |
| R3 | 尽量有中文注释 | 建立中文注释治理：表中文名、字段中文名、业务含义、取值说明、质量规则中文说明、AI 解释文本；缺失注释进入质控问题 |
| R4 | AI 对接系统后获取表关系，并根据各系统关联关系书写 SQL | 把 AI 工具从“目录”升级为“可执行只读工具”，提供系统级上下文、关系路径、字段检索、SQL 草稿、风险扫描、人工审核 |
| R5 | 人工复核和修改表字段关联关系和逻辑 | 建立关系工作台：候选关系、正式关系、字段映射、业务逻辑、审核状态、版本记录、变更审计 |
| R6 | 增加日常运维工具，给信息科处理补发、修复等日常问题 | 在当前平台内新增“运维工具箱”，按场景封装数据库操作/API 调用，强制权限、审批、审计和回滚说明 |
| R7 | 多系统用户与科室统一管理，由系统通过数据库处理 | 在当前平台内新增“统一用户与科室治理”模块：科室以 HIS `dept_dict` 为核心，人员以 HRP 人员表为正式主源、HIS 临时人员为补充，统一维护跨 HIS/EMR/无纸化/统一门户等系统账号/科室映射、同步任务和审计 |
| R8 | 增加基础字典维护，先搭框架，如诊断、手术等字典，考虑跨系统维护 | 在当前平台内新增“基础字典中心”，重点维护院内临床版、国家标准版、医保版诊断/手术三套编码及对照关系，发布后通过受控数据库同步到 HIS/EMR，兼容旧系统编码 |

## 2. 新总目标

平台目标从“只读资产门户”升级为：

1. **院内多系统数据资产目录**：清楚回答“有哪些系统、有哪些库、有哪些表、有哪些字段、每个字段中文含义是什么”。
2. **多系统数据质量检测平台**：清楚回答“哪个系统/库/表/字段有问题，问题严重程度、责任人、处理状态是什么”。
3. **表字段关系与业务逻辑复核平台**：清楚回答“这些表怎么关联，依据是什么，谁审核过，是否可信”。
4. **AI SQL 协作平台**：AI 能按系统上下文查表、查字段、查关系、生成 SQL 草稿，但不能绕过人工审核和安全边界。
5. **信息科日常运维工具平台**：把常见补发、重推、状态修复、接口重试等高频操作做成可审核、可追溯、可回滚的标准工具。
6. **院内多系统用户与科室统一管理平台**：科室以 HIS `dept_dict` 为核心，人员以 HRP 正式人员 + HIS 临时人员为来源，统一管理多个系统账号、人员/科室映射、角色权限、同步差异和同步结果，减少人工登录多个系统或直接改库的风险。
7. **诊断/手术字典中心**：优先治理院内临床版、国家标准版、医保版诊断/手术三套编码及对照关系，并维护版本发布、人工审核和同步到 HIS/EMR 的受控执行；药品、检验项目、检查项目等后续扩展。

## 3. 当前缺口

| 模块 | 当前状态 | 主要缺口 |
|---|---|---|
| 资产层级 | 当前以 `schema.table` 和业务域为主 | 不能清楚区分“系统/库/schema/表/字段”；HIS 源端 1234 表尚未合并展示 |
| 数据源管理 | 连接信息主要在文档里 | 没有结构化的数据源/系统表；不能对每个源库配置检测策略 |
| 质量检测 | 有 metadata_only 规则 | 还不是按系统/库/表/字段分层展示；缺少数据源连通性、采集状态、注释完整率、关系有效性总览 |
| 中文注释 | 字段注释来自元数据快照，缺失时只作为质量问题 | 没有人工维护中文名/业务含义/取值说明的工作台 |
| AI 工具 | `/api/v1/ai/tools` 已有 method/path/auth_required | 仍缺统一执行入口；AI 需要自己拼 HTTP 调用；上下文没有按系统/库分层组织 |
| 人工复核 | 候选关系可提升/拒绝 | 正式关系、字段映射、业务逻辑不能直接编辑；缺少版本、审计、复核原因 |
| 日常运维 | 当前平台没有运维操作工具 | 信息科仍需人工查库/改库/调用接口处理补发、重试、状态修复，风险高且不可追溯 |
| 用户与科室统一管理 | 当前平台没有跨系统用户/科室治理 | 科室以 HIS `dept_dict` 为核心但目标系统各自维护；正式人员在 HRP，临时人员在 HIS，电子病历/无纸化/统一门户等账号分散，缺少统一差异比对、映射、同步和审计 |
| 诊断/手术字典 | 当前只有业务术语维护和部分系统字典文档 | 院内临床版、国家标准版、医保版诊断/手术之间存在对照关系，缺少统一维护、人工审核、版本发布、同步审计和兼容旧编码能力 |

### P5.5：多数据库驱动与连接底座

> 目标：平台后续能够直接连接院内不同类型数据库，用于元数据采集、连通性检测、小样本质控和 AI SQL 草稿校验。驱动只负责连接与只读查询能力，不代表开放任意 SQL 执行。

| 数据库类型 | Python 驱动建议 | 备注 |
|---|---|---|
| Oracle | `oracledb` | 8.216 旧版 Oracle 必须支持 thick 模式，服务器需安装/配置 Oracle Instant Client；旧版 SQL 不支持 `FETCH FIRST`，统一用 `ROWNUM <= N` |
| SQL Server | `pyodbc` + Microsoft ODBC Driver 17/18 | 需离线准备 ODBC Driver 安装包；连接串支持 `TrustServerCertificate`、域账号/SQL 账号两种模式 |
| MySQL | `pymysql` 或 `mysqlclient` | 首选 `pymysql`，纯 Python 更适合离线部署；如性能要求高再评估 `mysqlclient` |
| PostgreSQL | `psycopg[binary]` | 当前应用库已使用 PostgreSQL，可复用连接池与 SQLAlchemy 方言 |
| 海量数据库 | 待用户提供驱动包与连接说明 | 需确认是否兼容 PostgreSQL 协议、JDBC/ODBC/Python 原生驱动、连接串格式、只读账号权限 |

### P11：信息科日常运维工具箱

> 可行性结论：建议集成到当前系统，不建议单独新建系统。原因是当前平台已经有数据源、资产目录、鉴权、审计、AI 工具、前端框架，运维工具正好复用这些底座。但执行层必须独立隔离，不能把“运维操作”混进普通资产查询或 AI SQL 能力里。

适合纳入的典型场景：

| 场景 | 示例 | 推荐实现方式 |
|---|---|---|
| 消息补发 | 某患者 HIS 未发送到电子病历，需要重新补发 | 封装为“补发任务模板”，输入患者标识/就诊号/业务类型，后台查状态后调用数据库过程或接口 |
| 接口重试 | HL7/ESB/EMR 消息失败重推 | 查询失败队列或日志表，按白名单接口重试 |
| 状态修复 | 某业务状态卡住，例如医嘱/检验/检查状态异常 | 只能做预定义模板，不开放任意 UPDATE |
| 医嘱修改 | 科室日常频繁处理的医嘱状态、医嘱属性、执行状态修正 | 必须做成场景化模板：输入患者/就诊/医嘱号，先只读核查影响范围，再按白名单 SQL/SP/API 执行；高风险必须审批和二次确认 |
| 消息推送/重推 | HIS 到电子病历、无纸化、统一门户等系统消息未推送或状态异常 | 查询消息队列表/接口日志，按业务单号触发重推或补偿，禁止手工拼接任意 SQL |
| 主数据同步辅助 | 科室/人员/诊断/手术字典同步失败后人工重试 | 只允许调用 P12/P13 已登记的同步模板或存储过程，不在 P11 另建一套字典/人员同步逻辑 |
| 数据核查 | 查询某患者某次就诊是否已同步 | 只读查询模板，输出脱敏核查结果 |
| 对账辅助 | HIS 与 EMR/LIS/PACS 某类数据数量或状态不一致 | 只读对账任务，生成差异清单，不自动改数据 |

禁止直接开放的能力：

- 禁止页面输入任意 SQL 后直接执行。
- 禁止 AI 自动执行 DML/DDL。
- 禁止无审批 UPDATE/DELETE/INSERT 生产业务库。
- 禁止把患者姓名、身份证、电话、地址等敏感明细暴露给外部 AI。

### P12：多系统用户与科室统一管理

> 可行性结论：建议集成到当前系统，但作为独立“身份与账号治理”模块。它与资产目录、数据源连接、运维工具共用底座，但权限必须比资产查看更严格。第一阶段先做“统一视图 + 映射 + 审计”，第二阶段再做“账号启停/角色同步”。

建设目标：

- 统一看见某个员工在 HIS、电子病历、移动护理、LIS、PACS 等系统中的账号。
- 维护员工主数据与各系统账号映射。
- 支持账号启用、停用、角色调整、科室调整的申请和同步任务。
- 所有账号变更必须审批、审计、可回溯。

#### P12 业务主线（用户已确认）

> 当前医院系统异构严重，人员与科室不是从零新建主数据，而是以现有业务库为核心源头，平台做“采集、比对、治理、同步、审计”的统一操作台。

| 对象 | 权威/核心来源 | 补充来源 | 同步目标 | 处理方式 |
|---|---|---|---|---|
| 科室/病区/护理单元 | HIS 库 `dept_dict` 表 | 其他系统已有科室表仅作对照和差异校验 | 电子病历、统一门户、无纸化等系统 | HIS 自行维护完成后，平台定期读取 `dept_dict`，与本地基线及目标系统对比，生成差异；支持自动/人工审批后通过数据库白名单模板同步到目标系统 |
| 正式人员 | HRP 系统人员表 | 目标系统账号表仅作账号状态/角色补充 | HIS、电子病历、无纸化、统一门户等系统 | 以 HRP 人员表为人员主源，平台定期采集 HRP 人员变化，与本地人员基线和各目标系统账号表比对，生成新增/停用/转科/角色差异，再通过白名单模板直接写目标库 |
| 临时人员 | HIS 系统维护的临时人员表/人员字典 | HRP 无记录但 HIS 有业务需要的临时账号 | 电子病历、无纸化、统一门户等需要临时人员的系统 | 作为 HRP 主源的补充来源，单独标识 `person_type='temporary'` 和 `primary_source_system='HIS'`，同步时不得覆盖 HRP 正式人员 |

关键原则：

- 科室以 `HIS.DEPT_DICT` 为核心，不在当前平台另造一套科室权威源；当前平台可预留科室维护入口，但维护动作本质是对 HIS/目标系统数据库的受控写操作，必须走 `asset_action_executors`。
- 人员以 HRP 人员表为正式人员基础，HIS 临时人员作为补充来源；平台本地只保存治理基线、映射、差异和审计，不替代 HRP/HIS 原系统维护职责。
- 定期查询/比对是核心触发动作：`HRP/HIS 源表 -> 本地基线 -> 目标系统表` 三方差异进入待处理清单。
- 所有直接数据库写入（HIS、电子病历、无纸化、统一门户等）都必须使用第 11 节定义的“运维写操作凭据”，并通过白名单 SQL/存储过程/API 模板执行。

### P13：基础字典中心

> 可行性结论：建议集成到当前系统，作为“数据治理 + 运维支撑”的基础模块。字典维护天然需要系统、数据源、表字段、跨系统映射、审计和发布能力，复用当前平台最合适。第一阶段先搭框架，不直接替代 HIS/EMR/LIS/PACS 原系统字典维护；先做统一视图、映射、版本、人工审核，再逐步支持同步到目标系统。

第一批建议纳入的字典类型：

| 字典类型 | 示例 | 主要系统 | 第一阶段建议 |
|---|---|---|---|
| 诊断字典 | 院内临床诊断、国家标准版诊断、医保版诊断 | HIS、EMR、医保、病案、质控 | **单独建诊断/手术对照表**，维护三套编码之间的一对多/多对多关系，维护后同步 HIS/EMR |
| 手术字典 | 院内临床手术、国家标准版手术、医保版手术 | HIS、EMR、手麻、医保、病案 | **单独建诊断/手术对照表**，重点解决院内名称、国家码、医保码不一致 |
| 科室字典 | HIS `dept_dict` 科室编码、病区、护理单元 | HIS、EMR、统一门户、无纸化 | 科室主线归 P12：以 HIS `dept_dict` 为核心源，P13 只在需要时做码表展示和跨系统编码对照 |
| 人员字典 | HRP 人员、HIS 临时人员、医生/护士/技师 | HRP、HIS、EMR、无纸化、统一门户 | 人员主线归 P12：以 HRP 人员表为正式人员基础，HIS 临时人员为补充，P13 不重复维护人员主数据 |
| 药品字典 | 药品编码、规格、厂家、医保编码 | HIS、药房药库、医保 | 二期纳入，复杂度高 |
| 检验项目字典 | 检验项目、组合项目、LIS 项目码 | LIS、HIS、EMR | 二期纳入 |
| 检查项目字典 | 检查项目、部位、影像分类 | PACS、HIS、EMR | 二期纳入 |

设计原则：

- **向上扩展**：支持国家标准码、省平台码、医保码、院内统一码、业务主题分类。
- **向下兼容**：保留各系统原始编码、原始名称、历史有效期、旧编码映射，不强行覆盖源系统。
- **跨系统映射**：一个标准字典项可对应多个系统字典项；同名不同码、同码不同名必须可识别。
- **版本发布**：字典变更先进入草稿/待审，通过后形成版本；是否同步到源系统另行审批。
- **安全同步**：同步到 HIS/EMR/LIS/PACS 等系统时只能走白名单 SQL、存储过程或接口，不能任意改库。

#### P13 诊断/手术字典业务主线（用户已确认）

诊断和手术字典不以某一个业务系统表作为唯一源，而是维护三套编码体系及其对照关系：

| 编码体系 | 说明 | 维护定位 |
|---|---|---|
| 院内临床版 | 医院临床实际使用的诊断/手术名称和编码，可能来源于 HIS/EMR/病案等系统 | 平台维护的院内临床口径，是与医生使用习惯最贴近的一套 |
| 国家标准版 | 国家标准诊断/手术编码，如 ICD-10、ICD-9-CM-3 等 | 作为标准化、上报、质控、病案统计的重要依据 |
| 医保版 | 医保诊断/手术编码或医保结算/贯标要求中的版本 | 作为医保结算、医保审核、支付方式改革相关口径 |

处理原则：

- 诊断/手术字典需要**单独建表**维护，不简单套用普通 `asset_dict_standard_items`。
- 三套编码之间允许一对一、一对多、多对多映射；同名不同码、同码不同义必须进入人工复核。
- 平台维护完成后，需要分别同步到电子病历系统、HIS 系统；同步方式是直接数据库操作，但必须通过 `asset_action_executors` 的白名单模板、审批、审计和运维写操作凭据执行。
- 一期先做维护与对照、差异清单、人工审核、受控同步；不做自动根据 AI 建议直接改库。

## 11. 安全边界

> 结构决策（详见第 20 节）：平台使用**两套凭据、两套执行路径**，避免"探库能力"和"运维写操作"混用同一账号导致误操作或权限扩散。

### 11.1 两套凭据

| 凭据类型 | 用途 | 权限 | 谁可以用 |
|---|---|---|---|
| **只读探库凭据**（`asset_data_sources.credential_ref` 默认类型） | P5.5 连接工厂、P6 系统分层采集、P8 质量检测、F46(P14) 元数据变更检测、AI SQL 协作(P10) | 仅 `SELECT`，禁止任何 DML/DDL；连接超时、行数限制、大表禁全扫 | 平台后端服务、AI 工具（AI 永远只能拿到这一套） |
| **运维写操作凭据**（`asset_data_sources` 另建 `write_credential_ref` 字段，或独立配置表，按数据源单独维护） | P11 运维工具箱、P12 人员/科室同步、P13 诊断/手术字典同步的白名单写操作 | 仅限执行预定义模板/存储过程/API，不允许拼接任意 SQL；权限范围以"能完成该模板"为最小化原则单独申请 | 仅 `asset_action_executors` 执行器内部使用，人工和 AI 都不能直接持有此凭据 |

### 11.2 两套执行路径

| 执行路径 | 覆盖模块 | 关键约束 |
|---|---|---|
| **只读探查路径** | P6 系统分层、P7 中文注释治理（读取部分）、P8 质量检测、P9 关系复核（读取部分）、P10 AI SQL、P14(F46) 元数据变更检测 | 使用只读探库凭据；AI 生成的 SQL 只能保存草稿，不自动执行；SQL 沙箱上线前不开放任意 SQL 执行 |
| **受控写操作路径** | P11 运维工具箱、P12 人员/科室/账号变更执行、P13 诊断/手术字典同步执行 | 必须走 `asset_govern_change_requests`（或 P11 专用 `asset_ops_tool_runs`）申请 → 审批（`approved_by != requested_by`，见 S02）→ `asset_action_executors` 白名单执行（模板/SP/API，见 S03）→ 写入 `asset_govern_audit_logs`（脱敏后，见 S05） |

### 11.3 通用安全规则

1. 源库默认只读；任何写操作必须显式使用运维写操作凭据，且只能通过白名单模板/存储过程/API 执行，不允许拼接任意 SQL。即使业务上确认“直接数据库同步/直接数据库操作”，实现上也必须通过 `asset_action_executors` 间接执行，不能开放裸 SQL。
2. 默认只做元数据与小样本检测；大表禁止全表扫描。
3. AI 只能调用只读探库凭据下的工具（元数据/关系/质量/只读采样），不能调用任何写操作执行器（见 S04）。
4. AI 生成 SQL 只保存草稿，不自动执行；人工审核通过后才可能进入受控 SQL 沙箱（暂不开放）。
5. 患者姓名、身份证、电话、地址、明细数据不得进入 AI 上下文；进入审计日志前必须脱敏（见 S05）。
6. 所有人工修改关系/注释/逻辑，以及所有运维/账号/字典写操作，必须写入统一审计表 `asset_govern_audit_logs`。
7. 审批流必须防自审：同一人不能既是 `requested_by` 又是 `approved_by`（见 S02）。
8. 白名单本身的新增/修改/禁用也要走审批流，不能运行时动态放开（见 S03）。

### 16.6 几个明确的"必须先补"建议（已在第 20 节固化为正式决策，此处保留原始建议供追溯）

1. **先建 RBAC 模型**：`asset_roles` + `asset_user_roles` + `asset_role_permissions`，让后续 P11/P12/P13 都引用同一套权限。
2. **先建统一审批/变更请求抽象**：`asset_govern_change_requests` 通用表 + 模块字段 `module/entity_type`，避免每模块各自建 change_request（P11 因执行细节强绑定例外，见 P11 节说明）。
3. **先建统一审计服务**：一张 `asset_govern_audit_logs` + 模块/动作/实体/前后值/operator，所有模块写同一表，前端按模块过滤。
4. **先建统一执行器抽象**：`asset_action_executors` 注册表（按 `execution_mode` 分发），P11/P12/P13 都注册到同一执行框架。
5. **先建连接凭据加密策略**：明确用 pgcrypto 还是外部 vault；`asset_data_sources` 加 `credential_ref` 不存明文；区分只读探库凭据与运维写操作凭据两套。
6. **不做 schema 划分**：统一在 `asset` schema 内用 `asset_ops_*/asset_identity_*/asset_dict_*/asset_govern_*` 前缀区分模块（已在第 20 节确认，不再拆分物理 schema）。
7. **先定义扩展点契约**：加新模块/新字典类型/新执行方式要走什么注册流程，写成 checklist 文档。
8. **迁移纪律**：明确手写 Alembic 迁移、不依赖 autogenerate；每张新表必须有 downgrade。

## 18. 元数据变更检测（F46：数据表字段扩展/修改/删除后的核查获取）

> 你新增的需求：当源系统数据表字段扩展、修改、删除后，平台能通过定时查询或其他方式核查并获取变更。
>
> 本功能属于数据资产治理的核心底座能力，必须复用 P5.5 的多数据库连接底座和 P6 的系统/数据源分层，不能单独做一套采集逻辑。

### 18.1 目标

- 自动发现院内各系统、各数据源中表和字段的变化。
- 变更类型包括但不限于：表新增、表删除、字段新增、字段删除、字段类型变更、字段长度/精度变更、字段可空性变更、字段注释变更、主键/索引变化（可选）、表行数显著变化（可选）。
- 支持定时自动检测和手动触发检测。
- 与质量模块联动：变更事件可自动进入问题台账，提示人工复核。
- 与中文注释治理联动：字段注释变更自动触发注释完整率重算。
- 与关系复核联动：字段删除或类型变更可能影响已有关系，需告警。

### 18.2 变更类型与处理策略

| 变更类型 | 说明 | 默认处理方式 |
|---|---|---|
| table_added | 源库新增表 | 标记为 candidate，进入待确认列表，不自动纳入正式资产 |
| table_removed | 源库删除表 | 标记为 deprecated，保留历史记录，人工确认后归档 |
| column_added | 表新增字段 | 进入变更事件，提醒人工补中文注释和业务说明 |
| column_removed | 表删除字段 | 高风险，进入 `asset_quality_findings` 并通知关系复核 |
| column_type_changed | 字段类型变更 | 高风险，进入 `asset_quality_findings` 并通知关系复核 |
| column_length_changed | 字段长度/精度变更 | 中风险，进入变更事件 |
| column_nullable_changed | 字段可空性变更 | 中风险，进入变更事件 |
| column_comment_changed | 字段注释变更 | 低风险，更新现有注释并记录变更 |
| index_changed | 主键/索引变化 | 可选，二期纳入 |
| row_count_changed | 表行数显著变化 | 可选，纳入质量指标，不触发元数据变更 |

### 18.3 数据模型

```sql
-- 已存在的 asset_metadata_snapshots 只保存表级摘要，需要新增列级快照表
-- 命名遵循第 20 节结构决策：统一 asset_ 前缀，不新建 schema
CREATE TABLE asset.asset_metadata_column_snapshots (
    id BIGSERIAL PRIMARY KEY,
    snapshot_id BIGINT NOT NULL REFERENCES asset.asset_metadata_snapshots(id),
    system_code TEXT NOT NULL,
    source_code TEXT NOT NULL,
    namespace_name TEXT,
    table_name TEXT NOT NULL,
    column_name TEXT NOT NULL,
    data_type TEXT,
    length INTEGER,
    nullable TEXT,
    column_default TEXT,
    comment TEXT,
    is_primary_key BOOLEAN DEFAULT false,
    UNIQUE(snapshot_id, system_code, source_code, namespace_name, table_name, column_name)
);

-- 变更事件表（与 asset_quality_findings、asset_govern_audit_logs、asset_govern_events 联动）
CREATE TABLE asset.asset_metadata_change_events (
    id BIGSERIAL PRIMARY KEY,
    snapshot_id_from BIGINT,
    snapshot_id_to BIGINT NOT NULL,
    system_code TEXT NOT NULL,
    source_code TEXT NOT NULL,
    namespace_name TEXT,
    table_name TEXT NOT NULL,
    column_name TEXT,
    change_type TEXT NOT NULL,          -- table_added/table_removed/column_added/column_removed/...
    before_value JSONB,
    after_value JSONB,
    severity TEXT DEFAULT 'info',       -- info/low/medium/high/critical
    status TEXT DEFAULT 'open',         -- open/acknowledged/resolved/ignored
    assigned_to TEXT,
    reviewed_by TEXT,
    reviewed_at TIMESTAMPTZ,
    review_note TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 定时任务实例：复用 P5.9-T11 统一调度服务 asset_scheduler_jobs 登记具体运行记录
-- 不再单独建 metadata_collection_jobs，改为 asset_scheduler_jobs 的 job_type='metadata_scan' 记录
CREATE TABLE asset.asset_scheduler_jobs (
    id BIGSERIAL PRIMARY KEY,
    job_type TEXT NOT NULL,             -- metadata_scan/identity_sync/dict_sync/quality_scan/...
    source_code TEXT,
    trigger_mode TEXT NOT NULL,         -- scheduled/manual
    schedule_cron TEXT,
    status TEXT DEFAULT 'pending',      -- pending/running/success/failed/cancelled
    started_at TIMESTAMPTZ,
    finished_at TIMESTAMPTZ,
    result_ref TEXT,                    -- 关联结果，如 snapshot_id、sync_batch_id
    total_processed INTEGER,
    total_changes INTEGER,
    error_message TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

> `asset_scheduler_jobs` 是第 20 节统一调度服务（P5.9-T11）的通用运行记录表，F46 的元数据采集任务只是 `job_type='metadata_scan'` 的一种；P12/P13 的定时同步任务复用同一张表，不再各自建调度表。

### 18.4 检测流程

```text
1. 配置采集任务（系统/数据源/扫描范围/频率/变更规则忽略项，规则存 `asset_change_rules`，见 P5.9-T13）
2. 触发采集（`asset_scheduler_jobs` 定时 / 手动按钮 / 外部 webhook）
3. db_connector 连接源库（只读采集凭据，见第 11 节安全边界）
4. adapter 采集 schema/owner、表、字段、类型、长度、注释、主键
5. 生成新的 snapshot（`asset_metadata_snapshots` 表级摘要 + `asset_metadata_column_snapshots` 列级快照）
6. 与上一次 snapshot 做 diff
7. 生成 `asset_metadata_change_events`
8. 根据 `asset_change_rules` 判断 severity
9. 进入变更事件列表 / `asset_quality_findings` / 统一事件流 `asset_govern_events`（P5.9-T12）/ 发送通知
10. 前端展示变更总览、按系统钻取、单条确认/忽略
```

### 18.5 任务规划

| 任务 | 内容 | 验收标准 |
|---|---|---|
| P14-T1 | 扩展快照模型到列级 | `asset_metadata_column_snapshots` 表可用，能保存每个字段的元数据快照 |
| P14-T2 | 元数据 diff 引擎 | 能对比两次快照并生成 table_added/table_removed/column_added/column_removed/type_changed 等事件 |
| P14-T3 | 变更事件模型 | `asset_metadata_change_events` 支持 severity/status/assigned_to/review 字段 |
| P14-T4 | 定时采集调度 | 接入 P5.9-T11 统一调度服务 `asset_scheduler_jobs`，支持按数据源配置 cron，夜间执行，失败告警 |
| P14-T5 | 手动触发采集 | 前端/接口可手动触发某数据源的元数据采集 |
| P14-T6 | 变更事件前端 | 变更事件列表、按系统筛选、确认/忽略/分配责任人 |
| P14-T7 | 与质量模块联动 | 高风险变更自动进入 `asset_quality_findings` |
| P14-T8 | 与关系复核联动 | 字段删除/类型变更时，提示相关关系需要复核 |
| P14-T9 | 与中文注释联动 | 新增字段自动标记为"缺少中文注释"质量问题 |
| P14-T10 | 历史快照清理策略 | 接入 P5.9-T14 数据生命周期管理，支持保留最近 N 次快照或按时间归档 |

### 18.6 API 建议

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/v1/sources/{source_code}/collect-metadata` | 手动触发元数据采集 |
| GET | `/api/v1/sources/{source_code}/snapshots` | 快照历史 |
| GET | `/api/v1/metadata-changes` | 变更事件列表 |
| PATCH | `/api/v1/metadata-changes/{id}` | 更新变更事件状态/负责人/备注 |
| GET | `/api/v1/metadata-changes/summary` | 变更统计（按系统/变更类型） |
| GET | `/api/v1/metadata-changes/{id}/impact` | 查看变更影响的表/字段/关系 |

### 18.7 安全与性能约束

- 只读采集，禁止改源库。
- 大表只做元数据，不做行数/全量扫描。
- 定时任务避开业务高峰，建议夜间执行。
- 采集超时可控，失败重试最多 3 次。
- 变更事件默认按系统/数据源降噪，避免同一字段反复抖动产生噪音。
- 快照数据量增长快，必须制定保留策略。

## 20. 结构决策记录（复核后固化，后续 AI 必须遵守，不得重新讨论）

> 本节由 2026-07-04 二次复核后新增，是对第 16/17/19 节提出的结构性问题的**最终决策**，不是又一轮建议。后续 AI 开发前必须先读本节；如需推翻本节决策，必须先征得用户确认，不能自行改回旧方案。

### 20.1 Schema 策略（已确认，不再讨论）

**决策：不拆分物理 schema，所有新表继续建在 `asset` schema 下。**

- 用户已明确选择"保持单一 `asset` schema"，原因：与当前代码库 865+ 张表体系一致、Alembic 迁移简单、SQLAlchemy `Base`/权限管理不需要跨 schema 处理；后续如确有必要再拆分也容易迁移。
- 用 `asset_<模块>_<实体>` 前缀区分模块，模块前缀固定为：
  - `asset_`：原有资产核心（`asset_tables`/`asset_columns`/`asset_relations`/`asset_systems`/`asset_data_sources` 等）
  - `asset_ops_`：P11 运维工具箱（如 `asset_ops_tool_templates`、`asset_ops_tool_runs`）
  - `asset_identity_`：P12 用户与科室统一管理（如 `asset_identity_departments`、`asset_identity_persons`、`asset_identity_person_sources`、`asset_identity_accounts`、`asset_identity_sync_diffs`）
  - `asset_dict_`：P13 基础字典中心（如 `asset_dict_categories`、`asset_dict_standard_items`、`asset_dict_medical_code_sets`、`asset_dict_medical_code_items`、`asset_dict_medical_code_mappings`、`asset_dict_medical_sync_diffs`）
  - `asset_metadata_`：F46/P14 元数据变更检测（如 `asset_metadata_column_snapshots`、`asset_metadata_change_events`）
  - `asset_govern_`：跨模块治理底座（如 `asset_govern_change_requests`、`asset_govern_audit_logs`、`asset_govern_events`、`asset_govern_enum_values`）
  - `asset_quality_`：质量检测（沿用现状：`asset_quality_rules`/`asset_quality_findings`/`asset_quality_check_runs`/`asset_quality_tasks`/`asset_quality_metrics`）
  - `asset_scheduler_`：统一调度服务（`asset_scheduler_jobs`）
  - `asset_change_rules`：变更规则引擎（跨模块共用，不加模块前缀，因为规则本身覆盖多个模块）
- **后续 AI 不得**：新建 `ops`/`identity`/`dict`/`govern` 等物理 schema；不得把某个模块的表分散建到多个 schema。

### 20.2 凭据与执行路径策略（已确认，不再讨论）

**决策：两套凭据 + 两套执行路径**（详见第 11 节）。

- 只读探库凭据（`asset_data_sources.credential_ref`）：供探库/AI/质控/F46 元数据采集使用，只能 `SELECT`。
- 运维写操作凭据（`asset_data_sources.write_credential_ref`）：仅供 `asset_action_executors` 内部使用，人工和 AI 都不能直接持有，只能通过白名单模板/存储过程/API 间接触发写操作。
- **后续 AI 不得**：让 AI 工具、探库服务直接持有或拼接写操作凭据；不得让运维/账号/字典同步复用只读探库凭据。

### 20.3 跨模块底座canonical表清单（唯一权威命名，后续 AI 直接引用，不得另起名字）

### 20.4 原遗留问题的最终决策（用户已确认，后续 AI 不得重复提问）

以下三项曾在二次复核时列为开放项。用户已于本轮明确确认处理方式，后续 AI 必须按本表执行；如需推翻，必须再次征得用户确认。

| 编号 | 原问题 | 最终决策 | 执行要求 |
|---|---|---|---|
| E01 | 候选关系提升为正式关系时，是"复制一行"还是"引用同一行改状态" | **引用同一行改状态** | P9 工作台以 `asset_relation_reviews` 作为统一关系工作流主表。候选提升时更新同一行的 `relation_scope/review_status/validation_status`，不复制新行；历史 `asset_relations`/`asset_candidate_relations` 作为来源表保留，迁移时要能追溯来源 |
| F46-E04 | 字段变更是否自动联动业务术语/字典分类 | **一期只读提示，不自动写入** | F46 只生成提示、待办或 `asset_quality_findings`，不自动修改 `asset_business_terms`、`asset_dict_*` 或字段注释。需要写入时必须由人工在 P7/P13 工作台确认 |
| E02 | `execution_mode` 是否要做成真正插件系统 | **暂不做真插件，先枚举表 + dispatcher dict 过渡** | `execution_mode` 值纳入 `asset_govern_enum_values`；后端使用 dispatcher dict 分发到具体执行函数；只有当 execution_mode 种类明显膨胀或第三方执行器接入时，再评估真正插件化 |

后续 AI 不得把这三项重新列为"需要用户确认"，除非它提出的实现方案会推翻上表决策。

### 20.5 人员、科室、诊断/手术字典的业务源头决策（用户已确认）

> 本节是对 P12/P13 的业务口径补充。后续 AI 不得把人员、科室、诊断/手术字典都做成平台自造主数据；必须尊重医院现有业务库的权威来源和直接数据库同步现状。

| 对象 | 权威/核心来源 | 平台角色 | 同步方式 | 不得做什么 |
|---|---|---|---|---|
| 科室/病区/护理单元 | HIS 库 `dept_dict` | 平台采集 HIS `dept_dict` 形成本地治理基线，做跨系统对照、差异检测、同步审批和审计；当前系统预留维护入口，但维护动作本质是受控写 HIS/目标库 | 直接数据库同步到电子病历、统一门户、无纸化等系统；必须走 `asset_action_executors` + 运维写操作凭据 + 审批审计 | 不得另造一套脱离 HIS 的科室权威字典；不得页面裸 SQL 改目标库 |
| 正式人员 | HRP 系统人员表 | 平台以 HRP 人员表为正式人员基础，形成 `asset_identity_persons` 基线，做目标系统账号映射和差异检测 | 通过白名单数据库模板同步到 HIS、电子病历、无纸化、统一门户等系统 | 不得把 HIS 临时人员覆盖为 HRP 正式人员；不得由 AI 自动执行人员变更 |
| 临时人员 | HIS 系统维护的临时人员表/人员字典 | 平台作为 HRP 主源的补充采集，标识 `person_type='temporary'`、`primary_source_system='HIS'` | 需要时同步到电子病历、无纸化、统一门户等系统，同样走白名单执行器 | 不得与 HRP 正式人员混同，不得覆盖 HRP 人员状态 |
| 诊断/手术字典 | 平台维护三套编码体系：院内临床版、国家标准版、医保版 | 平台单独建 `asset_dict_medical_code_sets/items/mappings` 维护三套编码及对照关系 | 维护审核后直接数据库同步到 HIS、电子病历系统；必须走白名单执行器和审计 | 不得只用普通 `asset_dict_standard_items` 简化掉三套编码对照；不得 AI 建议后自动写库 |

定期触发方式：

1. P12 人员/科室：通过 `asset_scheduler_jobs` 定期查询 HRP 人员表、HIS `dept_dict`、HIS 临时人员表以及目标系统账号/科室表，生成 `asset_identity_sync_diffs`。
2. P13 诊断/手术：通过 `asset_scheduler_jobs` 定期查询 HIS/EMR 目标字典表，与平台三套编码基线比对，生成 `asset_dict_medical_sync_diffs`。
3. 所有差异默认只进入待处理清单；真正写目标库必须走 `asset_govern_change_requests` 或 `asset_ops_tool_runs` 审批流，再由 `asset_action_executors` 执行。
