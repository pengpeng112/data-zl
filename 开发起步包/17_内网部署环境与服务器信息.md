## 3. 连接方式

### 3.1 本机 → 跳板机 8.53（免密）

```bash
ssh -p 40022 -i ~/.ssh/id_ed25519_ai root@10.10.8.53
```

- 密钥：本机 `~/.ssh/id_ed25519_ai`（已验证可用）。
- 跳板机：openEuler 22.03，Python 3.9.9，无 paramiko/expect/sshpass，有 ssh-copy-id/pip3/dnf，**无外网**。

### 3.2 跳板机 → 8.83（SSH_ASKPASS 非交互密码登录）

跳板机无 sshpass/expect，利用 OpenSSH 的 `SSH_ASKPASS` 机制实现非交互密码登录（openEuler 的 OpenSSH 支持 `SSH_ASKPASS_REQUIRE=force`）。PowerShell 经 base64 传脚本执行，规避引号转义：

```powershell
# inner：在 8.83 上执行的命令
$inner = @'
echo OK; hostname
'@
$inner_b64 = [Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes($inner))
$outer = @"
printf "#!/bin/bash`necho <ROOT_PASS>`n" > /tmp/ap.sh
chmod +x /tmp/ap.sh
export SSH_ASKPASS=/tmp/ap.sh SSH_ASKPASS_REQUIRE=force DISPLAY=:0
setsid -w ssh -o StrictHostKeyChecking=no -o PreferredAuthentications=password -o PubkeyAuthentication=no -o ConnectTimeout=10 root@10.10.8.83 'echo $inner_b64 | base64 -d | bash' </dev/null 2>&1
"@
$outer_b64 = [Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes($outer))
ssh -o BatchMode=yes -o StrictHostKeyChecking=no -o ConnectTimeout=10 -p 40022 -i "$env:USERPROFILE\.ssh\id_ed25519_ai" root@10.10.8.53 "echo $outer_b64 | base64 -d | bash"
```

> `<ROOT_PASS>` 处替换为 8.83 root 实际密码（见 §4 凭据）。该 askpass 脚本 `/tmp/ap.sh` 一次性，可固化复用。

### 3.3 文件中转通道

```text
本机 ──scp -P 40022 -i id_ed25519_ai──► 8.53:/tmp/
8.53 ──scp（askpass 注入密码）──► 8.83:/opt/deploy/
```

跳板机到 8.83 的 scp 同样用 SSH_ASKPASS 注入密码（把 §3.2 的 `ssh` 换成 `scp`）。

## 6. 离线部署通道与约束（最关键）

**核心约束：全链路无外网**（跳板机、8.83 均不能联网），所有组件必须离线送达。

**离线包准备与送达路径**：

```text
[本机 Windows，有外网]
   1. 下载离线包：rpm/wheel/tar/镜像
   2. 本机构建 Vue 静态产物（npm build）
        │ scp -P 40022
        ▼
[跳板机 8.53，/tmp/ 中转]
        │ scp（askpass 注入密码）
        ▼
[8.83，/opt/deploy/ 落地]
   3. dnf localinstall / pip install --no-index / docker load / pg_ctl 配置
```

**离线包清单（待准备）**：

| 组件 | 离线形态 | 来源 | 目标 |
|---|---|---|---|
| pgvector | 源码 tar（PG14 编译）或 rpm | github/Anhv/pgvector | 8.83 编译安装到 PG |

## 7. 部署组件决策矩阵

基于"全链路无外网 + 已有原生 PG14 + 已有 Docker（无 compose）"的环境，组件部署方式初判：

| 组件 | 部署方式 | 依据 | 状态 |
|---|---|---|---|
| PostgreSQL 元数据库 | **复用原生 PG14，新建专用库** | 已装，避免容器版与原生版端口/数据冲突 | ⏭ 待建库 |
| pgvector | 离线源码编译安装进原生 PG | MVP 向量检索必需（`01` §5.3） | ⏭ 待装 |
| FastAPI 后端 | **venv + systemd**（非容器） | 离线 wheels 简单，直连原生 PG 与 8.216 | ⏭ 待部署 |
| 探库 Worker | venv + systemd/APScheduler | 与 FastAPI 共享 venv 或独立 | ⏭ 待部署 |
| Vue 管理端 | **本机构建 → 静态产物传 8.83** | 8.83 无 Node，避免离线装 Node | ⏭ 待构建 |

## 8. 与平台架构（`01`）的对应

本次部署承载 `01` 阶段一（MVP）+ 阶段二/三的服务：

```text
8.83 服务器
  ├─ PostgreSQL 元数据库 ── 对应 01 §5.3 元数据库 + §7 数据模型
  │     ├─ pgvector ─────── 对应 01 §5.3 向量库 MVP（RAG/字段语义）
  │     └─ 导入现有资产包 ── 对应 01 §16.17（08/09/资产包/03）
  ├─ FastAPI 后端 ──────── 对应 01 §6 全部功能模块 + §8 连接器
  │     └─ AI 工具 API ──── 对应 01 §6.8（供外网 AI 只读调用，写视图/分析 SQL）
  ├─ 探库 Worker ────────── 对应 01 §6.2 自动探库 + §16.3 负载治理
  │     └─ 直连 8.216 探 ODS；HIS 后续经跳板机
  ├─ Vue 管理端 ────────── 对应 01 §6.10 展示应用
  └─ 对接内网 Dify ─────── 对应 01 §6.7（不重复部署模型）
```

**外网 AI 协作模式**（用户重点诉求）：不让 AI 直连数据库，而是通过 FastAPI 暴露的**只读 AI 工具 API**（`search_tables` / `get_table_profile` / `find_relation_path` / `get_confirmed_relations` / `generate_sql` / `validate_sql`），AI 基于已确认关系图谱写视图、分析 SQL，再经 **SQL 沙箱**（`01` §16.4 四层防护）受控执行。既让 AI 方便分析，又不接触真实患者数据。
