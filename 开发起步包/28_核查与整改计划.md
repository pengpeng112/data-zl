## 2. 后端业务逻辑缺口（高优先级）

### 2.1 P5.5-T4：数据源连通性检测是占位桩

**现状**：`POST /api/v1/sources/{source_code}/check` 只返回 `"status": "pending"` 和 `"连通性检测执行器待 P5.5-T4 实现"` 的文本。

**整改**：需要真正调用 `DatabaseConnector.test_connectivity()`，读取 `AssetDataSource` 的 db_type/host/port 等字段，构建连接器实例并执行连通性测试，然后将结果写回 `last_check_status` / `last_check_at`。

### 2.2 P8-T4：小样本质检未真正连接源库

**现状**：质检规则的 `RULE_RUNNERS` 只做基于本地元数据的检查（comment 缺失等），不连接源库做小样本采样。

**整改**：在 `quality.py` 增加 `SAMPLE_ORPHAN_RATE` 规则的实际执行器，通过 `DatabaseConnector` 对白名单表执行 `ROWNUM <= N` 采样，计算孤儿率。

### 2.3 P11-T6：首批运维工具模板未接入

**现状**：`asset_ops_tool_templates` 表只有 CRUD API，没有预置工具模板（消息补发、重推、医嘱核查、状态修复）。

**整改**：在数据库初始化脚本/Alembic data migration 中插入至少 3 条模板记录：
- `msg_resend`：消息补发/重推（查询 + 白名单 SP）
- `order_status_check`：医嘱状态只读核查
- `sync_status_check`：数据同步状态核查

### 2.4 P12-T3/T4：人员科室采集与差异比对未实现

**现状**：`identity.py` API 只有 CRUD 查询，没有从 HIS/HRP 源库实际采集的逻辑，也没有自动差异比对。

**整改**：
- 实现 `POST /api/v1/identity/collect-sources`：连接 HIS 源库读 `dept_dict`、临时人员表；连接 HRP 读正式人员表
- 实现差异比对函数：`HRP源表 vs 本地 baseline vs 目标系统表` → 生成 `asset_identity_sync_diffs`

### 2.5 P13-T5：HIS/EMR 字典差异采集未实现

**现状**：`dict_medical_api.py` 只有 CRUD + 对照维护，没有从 HIS/EMR 实际读取字典表进行比对的动作。

**整改**：实现采集逻辑，连接 HIS/EMR 源库读取诊断/手术字典表，与 `asset_dict_medical_code_items` 比对，生成 `asset_dict_medical_sync_diffs`。

### 2.6 P14-T4：定时调度未接入 APScheduler

**现状**：`asset_scheduler_jobs` 表已建，API 只做手动记录。没有集成 APScheduler 的定时触发。

**整改**：在主应用启动时创建 `APScheduler` 单例，读取 `asset_scheduler_jobs` 中 `trigger_mode='scheduled'` 且 `schedule_cron` 非空的记录，注册为定时任务。

### 2.7 P5.9-T9：跨系统同步通用框架缺失

**现状**：计划要求 `sync_executor` 作为跨系统同步的通用框架，但当前只有 `asset_action_executors` 表注册，没有实际的 `sync_executor` 执行类。

**整改**：新建 `app/services/sync_executor.py`，实现：

- 结果写入 `asset_govern_audit_logs`（module='sync'）

### 2.8 P5.9-T14：数据生命周期管理策略缺失

## 3. 数据模型与架构合规问题（中优先级）

### 3.1 P9 关系复核缺 `asset_relation_reviews` 表

**现状**：计划明确要求 P9 使用 `asset_relation_reviews` 作为统一关系工作流主表（§5.3）。但当前代码中**不存在此表**——关系处理仍直接用 `AssetRelation`（`asset_relations` 表）做 CRUD。候选关系提升/拒绝也在 `candidates.py` API 里直接操作 `CandidateJoin`。

**影响**：
- 无法支持 "候选提升为正式时只改同一行状态，不复制新行"（E01 决策）
- 无法支持字段级映射管理
- 无法支持关系人工复核的 draft/reviewing/approved/rejected 状态机
- 无法支持 `relation_desc_cn`、`business_logic_cn`、`source_evidence` 等复核字段

**整改**：新建 `AssetRelationReview` 模型 + Alembic 迁移，迁移现有 `asset_relations` 数据到新表，调整 `relations.py` / `candidates.py` API 指向新表。

### 3.2 RBAC 中间件未使用

**现状**：`asset_roles` / `asset_user_roles` / `asset_role_permissions` 三张 RBAC 表已建模且有 CRUD API，但 `main.py` 的认证中间件仍是简单 Bearer Token 验证，不查 RBAC。

**整改**：在 `auth_and_logging_middleware` 中增加角色权限校验：
1. 从 Token 查到对应的 `user_identifier`
2. 通过 `asset_user_roles` 查到角色列表
3. 通过 `asset_role_permissions` 校验当前请求的资源+动作是否在权限列表中
4. 或者：先只加 RBAC 查询函数，前端菜单按角色过滤（阶段 1 简化方案）

### 3.3 `asset_ops_tool_runs` 审批状态与 `asset_govern_enum_values` 未同步

**现状**：枚举表 `asset_govern_enum_values` 已建，但 `ops_tool.py` API 中审批状态校验仍是硬编码 `("draft", "pending")`。

**整改**：将 ops API 中的审批状态校验改为从 `asset_govern_enum_values` 动态读取 `enum_code='approval_status'` 的合法值列表。

### 3.4 审计日志缺少脱敏

**现状**：`GovernAuditLog` 的 `before_data`/`after_data` 字段直接写入原始 JSON。计划 S05 决策要求写入前必须脱敏。

**整改**：在 `_write_audit()` 函数中增加 `mask_sensitive(data)` 调用，对 `姓名/身份证/电话/地址/MRN` 等敏感字段做脱敏处理（保留前1后1字符或 hash）。

### 3.5 运维审批防自审查询参数不一致

**现状**：`ops_tools.py` 的 approve 端点用 `Query(...)` 传 `approved_by`，但 `governance.py` 的 approve 端点用 `BaseModel` 传 `approved_by`。两个模块对防自审的实现方式不同。

**整改**：统一 `ops_tools.py` 的 approve 为 request body（`class ApproveRequest(BaseModel): approved_by: str`），保持与 `governance.py` 一致。

## 8. 对抗性测试发现的具体 Bug

对抗性测试 27 条全通过，但通过代码审查发现的潜在问题：

| 编号 | 问题 | 文件:行号 | 风险 |
|---|---|---|---|
| B1 | `metadata_diff.py` 的 `system_code` 在 table-level diff 时可能为空串 `""`，导致变更事件的 `system_code` 字段不一致 | `metadata_diff.py:37` | 变更事件查询时按 system_code 过滤会漏掉这些记录 |
| B2 | `metadata_changes.py` 中 `collect-metadata` 对不存在的 source_code 仍返回 200，生成空快照 | `metadata_changes.py:39-41` | 静默产生垃圾快照数据 |
| B3 | `ops_tools.py` 中 `approved_by` 使用 Query 参数传递，可能暴露在 URL 日志中 | `ops_tools.py:117` | 审计信息泄露（低风险，但建议改为 Body） |
| B4 | `governance.py` 的 `approve_change_request` 不校验 `requested_by` 是否为空，若请求未传 `requested_by` 则防自审失效（因为 None != None 是 False） | `governance.py:234` | 若 `requested_by=None` 且 `approved_by=None`，防自审会被绕过。**需要修复** |

### B4 详细说明（需要立即修复）

```python
# governance.py L234
if req.approved_by == cr.requested_by:
    raise HTTPException(status_code=400, detail="审批人与申请人不能为同一人")
```

当创建变更请求时未传 `requested_by`（字段是可选的 `str | None = None`），`cr.requested_by` 为 `None`。审批时若也传 `approved_by=None`（或前端未传该字段），则 `None == None` 为 `True`，会正确拦截。但若 `requested_by=None` 而 `approved_by="some_user"`，则不会拦截——但这并没有安全风险，因为申请人未标识时审批人不可能是同一人。实际风险是：**如果攻击者知道内部用户名，且创建请求不填 `requested_by`，审批不填 `approved_by`，两个 None 会被拦截。真正的风险在于运维端点的 `approved_by` 用 Query 参数传递**，导致空字符串 `""` 和 `None` 行为不一致。

**修复**：在审批端点增加 `if not req.approved_by or not cr.requested_by: raise HTTPException(400, "缺少操作人信息")`。
